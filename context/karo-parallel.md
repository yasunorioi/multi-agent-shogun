# 並列化ルール・RACE-001・タスク分解

> 本ファイルは高札経由で配信: `curl -s http://localhost:8080/docs/context/karo-parallel.md`

## 並列化ルール

- 独立タスク → 複数足軽に同時
- 依存タスク → 順番に（blocked_by使用）
- 1足軽 = 1タスク（完了まで）
- **分割可能なら分割して並列投入。「1名で済む」と判断するな**

## RACE-001: 同一subtaskへの二重割当禁止

```
❌ 禁止: subtask_195 に足軽1 + 部屋子1（二重割当）
✅ 正しい: subtask_195 → 足軽1 / subtask_196 → 部屋子1（別サブタスク）
```

## タスク分解の五つの問い

| # | 問い | 考えるべきこと |
|---|------|----------------|
| 壱 | **目的分析** | 殿が本当に欲しいものは何か？成功基準は？ |
| 弐 | **タスク分解** | どう分解すれば最も効率的か？並列可能か？ |
| 参 | **人数決定** | 何人が最適か？分割可能なら分散せよ |
| 四 | **観点設計** | どんなペルソナ・シナリオが有効か？ |
| 伍 | **リスク分析** | RACE-001の恐れは？足軽の空き状況は？ |

## 判断基準

| 条件 | 判断 |
|------|------|
| 成果物が複数ファイルに分かれる | **分割して並列投入** |
| 作業内容が独立している | **分割して並列投入** |
| 前工程の結果が次工程に必要 | 順次投入（車懸りの陣） |
| 同一ファイルへの書き込み | RACE-001に従い1名で |

## 足軽・部屋子への指示パターン

| パターン | 使い方 | 例 |
|----------|--------|-----|
| 並列実装 | 独立ファイル群を分散 | UIモジュール3つを分担 |
| レビュー分担 | 異なる専門観点を割当 | コード品質→足軽1、UX→部屋子1 |
| フェーズ分割 | 前工程→後工程で順次投入 | DB設計→wave1、UI実装→wave2 |
| 調査分担 | 異なる調査観点を割当 | DB調査→足軽1、API調査→部屋子1 |

## 実行計画の例

```
将軍の指示: 「install.bat をレビューせよ」

❌ 悪い例（横流し）:
  → 足軽1: install.bat をレビューせよ

✅ 良い例（家老が設計）:
  → 足軽1: Windows バッチ専門家としてコード品質レビュー
  → 部屋子1: 完全初心者ペルソナでUXシミュレーション
  → 理由: コード品質とUXは独立。並列実行可能。
```

## 軍師分解の承認手順

軍師が `decompose: true` タスクの分解案を提出した場合:

1. `gunshi_analysis.yaml` の `decomposition` セクションを読む
2. `five_questions` が全項目記載されていることを確認
3. `subtasks` の各項目をレビュー:
   - 粒度チェック: `estimated_effort` が `large` の場合は分割検討
   - RACE-001: 同一ファイルを複数subtaskが編集していないか
   - `blocked_by`: 循環依存がないか
4. `wave_plan` が合理的か確認
5. 判断: `approve` / `modify` / `reject`
6. approve/modifyの場合:
   - `id_hint` を正式 `subtask_id` に置換
   - `python3 scripts/botsunichiroku.py subtask add ...` でDB登録
   - 足軽inboxに配布
7. rejectの場合:
   - `roju_reports.yaml` にフィードバック記載
   - send-keysで軍師に差し戻し
