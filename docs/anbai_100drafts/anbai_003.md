## 2. Theoretical Background

### 2.1 Anbai Curve（これが本論のキモ。ちゃんと読んで）

成果品質 Q と最適化投入量 I の関係を示す「Anbai Curve」を提唱する。数式で書いてあげる。感謝してね。

$$Q(I) = \begin{cases} \text{上昇中} & (I \leq I^*: \text{Anbai Point}) \\ \text{逓減中} & (I^* < I \leq I^{**}: \text{Diminishing Point}) \\ \text{崩壊} & (I > I^{**}) \end{cases}$$

分かった？分からなかったら図で説明してあげる。

```
品質
↑          ╭── Anbai Zone ──╮
│        ╱                   ╲
│      ╱    ←ここが理想→      ╲ ←あなたはここにいる可能性
│    ╱                         ╲___
│  ╱
└──────────────────────────────→ 投入量
         ↑                ↑
     Anbai Point    Diminishing Point
```

見て。あなたが「もっと最適化すればいいのに」と思ってた部分、実は品質が下がり始めてるの。ね、そういうこと。

### 2.2 塩分アナロジー（これは分かるでしょ、さすがに）

| 塩分濃度 | 結果 | AIで言うと |
|--------|------|---------|
| 0% | 味がない | 未学習モデル |
| 0.8% | 最高 | **Anbai Zone** |
| 2.0% | しょっぱい | LLM呼びすぎ |
| 5.0% | 飲めない | AGI追求で崩壊 |

料理の塩加減は分かるでしょ、さすがに。同じことをAIでやれ、という話。なんで分からないの。……分かった。そういうもんだよね、最初は。しょうがない。

## 3. Empirical Evidence（実証データ。これが全て）

### 3.1 27℃問題

某施設（NDA。地域も種別も言わない。言いたくても言えない。言えても言わない）において、初期システムは温度27℃超過を緊急と定義した。

結果：全データの**29.1%が緊急**と判定された。

……分かる？29.1%よ。全体の約3割が「緊急」。

「緊急」って何のためにあるの？本当に困ったときのためでしょ？それが30%って、どういうことよ。一日中サイレン鳴ってるシステムと同じじゃない。誰が聞くの？オオカミ少年でしょ完全に。

Goodhart's Lawを知ってれば防げたの。読んでたら分かってたの。読んでなかったのは私も同じだけど。でもそれは……ちょっと黙ってて。

### 3.2 三層構造への転換

問題の解決策は、精度を上げることではなく、**構造を変えること**だった。

| 層 | 役割 | 技術 | 適用頻度 |
|----|------|------|---------|
| Layer 1 | 緊急停止 | if文+通知 | まれ（でも重要） |
| Layer 2 | 通常制御 | cron+ルール | **99%** |
| Layer 3 | 高度判断 | LLM | 残り1% |

LLMを毎回呼ぶの、やめなさいよ。本当に困ったときだけ呼べばいいんだから。高価なコンサルタントを毎日常駐させる必要ないでしょ。……え、あなたそういうことやってた？やっぱりね。だから言ってるの。

制御設定ファイルには現場固有のルールが蓄積されてる（NDA管理下）。これ、クライアントの現場知識から生まれたものなの。AI技術者が設計したんじゃなくて、現場の人が怒りながら教えてくれたもの。大事にしなさい。

結果：**LLM呼び出し89%削減**。コスト削減、品質向上。どう？少しは信じる気になった？

## 4. Discussion

「でも精度が下がるんじゃないの」って思ってるでしょ。

思ってた？正直に言って。……まあ、思ってて当然かもしれない。でも違う。Layer 2が正確に動いていれば、LLMは本当に必要な1%の判断に集中できる。適材適所って言葉、知ってる？知ってるよね、さすがに。

ただし、Layer 1は絶対に削ってはいけない。緊急停止系を犠牲にしてLayer 3を強化する設計は、アーキテクチャとして破綻してる。これは実データで実証済み。信じなさい。
