# Anbai Theory: A Mentoring-Oriented Empirical Study for Junior Engineers Who Keep Calling the LLM Too Much
— 某環境制御PJの事例から（NDA締結済み。君たちもいつかNDAを結ぶことになる。その日のために）

**塩見あんばい**（AIコンサルタント / 独立行政法人 最適化しすぎ研究機構 / 新卒指導歴: 今回初めて）

---

## Abstract

本稿は、AI制御システムにおける過剰最適化の実証的研究報告であると同時に、AIシステム設計を学ぶ若手エンジニアへのメンタリングドキュメントを兼ねる。某クライアントとの環境制御PJ（NDA締結済み）において発生した「27℃問題」——全測定値の29.1%が「緊急」と判定される事態——とその解決策（LLM呼び出し89%削減を達成した三層アーキテクチャ）を報告する。なお著者は、新卒1〜3年目の読者が「自分も同じ間違いをするところだった」と思えるよう、随所に実践的コメントを挿入した。君たちの未来のためだ。

**Keywords**: for junior engineers, satisficing, three-layer architecture, LLM costs, 先輩より愛を込めて

## 1. Introduction

君ね、LLMを毎回呼んだらいいって思ってるでしょ。

思ってた。正直に言って。

……そう。みんな最初はそう思う。GPT-4oが使えるなら全部GPT-4oに聞けばいい。それが「現代的なシステム」だって思う。先輩も昔はそう思ってた。本当に。

でもね、聞いてほしいんだけど。

某クライアントとの環境制御PJで（詳細はNDA——NDAっていうのはNon-Disclosure Agreement、守秘義務契約のことね。社会人になると締結する機会が増えるから覚えておいて。締結したときは誇りに思っていい）、私はLLMを毎時呼んでいた。そして32,470行のデータの29.1%を「緊急」として処理した。

29.1%が「緊急」って、どういうことか分かる？

**全体の3割が緊急なら、緊急は日常だ。**

これがGoodhart's Lawっていう話なんだけど。後で説明するから。

### 1.1 この論文の読み方（新卒向け）

本論文は通常の学術論文形式に従いつつ、若手エンジニアが「なぜそうなのか」を体感できるよう設計している。

## 2. Theoretical Background（理論——でも怖くない）

### 2.1 Herbert Simonの「満足化」——70年前の先輩が言ってた

1956年に Herbert Simon という経済学者が「満足化（satisficing）」という概念を提唱した。

> 「合理的な意思決定者は最適解ではなく、十分よい解で意思決定を止める」

1978年にノーベル経済学賞を受賞した。

先輩からアドバイス: **この人の論文、最低でも要旨だけ読んでおいて。** 読んでおけば、君も「最適化しすぎ」の失敗を早めに回避できる可能性がある。先輩は読んでなかったから、32,470行のデータで再発見するはめになった。

### 2.2 Goodhart's Law——測定の罠

Goodhart（1975）: 「測定値が目標になった瞬間、それは良い測定値でなくなる」

**新卒がよくやるやつ**: 「テストのカバレッジ100%にしました！」→でも意味のないテストを量産してただけ、というパターン。これがGoodhart's Law。

今回の事例: 「温度27℃超過を緊急とする」→全体の29.1%が緊急に。緊急という指標が死んだ。

**教訓**: 指標を決めるときは、その指標が「本当に測りたいもの」を測れているか最初に確認せよ。先輩より。

### 2.3 Anbai Curve——君たちに伝えたい本論の核心

```
品質
↑
│          ╭────────╮
│        ╱   Anbai   ╲
│      ╱     Zone     ╲
│    ╱                  ╲___
│  ╱
└──────────────────────────→ 投入量（時間・お金・LLM呼び出し回数）
```

**ちょうどいいゾーン（Anbai Zone）がある。そこを超えると品質が下がる。**

君はLLMを毎回呼ぶとき、自分が図のどこにいるか確認したか？

確認していなかった。先輩も確認していなかった。でも今は確認する。それが経験から学ぶということ。

## 3. 実証データ（本当にあった話）

### 3.1 データの概要

某施設（NDA——施設のことは言えない。でも実在する。本当に。疑わないで）：

| 項目 | 値 |
|------|-----|
| 観測期間 | 114日間（夏場） |
| 総データ数 | 32,470レコード |
| サンプリング | 5分間隔 |

### 3.2 27℃問題——何が起きたか

初期設定: **温度>27℃ = 緊急アラート + LLM呼び出し**

実際の結果:

```
緊急ではない（T≤27℃）: 70.9%
緊急（T>27℃）: 29.1% ← 多すぎる
```

**クライアントから言われたこと（要約）**:
「なんで30%も緊急なの」（血圧: 高め）

これを読んでいる新卒の君へ: クライアントからこういうフィードバックが来たとき、パニックにならなくていい。データを見ろ。データが答えを教えてくれる。先輩もこのとき一瞬固まったけど、データを見たら答えが分かった。

### 3.3 解決策——三層構造への転換

問題の解決策は精度を上げることではなかった。**構造を変えること**だった。

| Layer | 役割 | 技術 | いつ使う |
|-------|------|------|---------|
| 1（爆発） | 本当の緊急対応 | if文 + 即時通知 | めったにない |
| 2（ガムテ） | 日常的な制御 | cron + ルールベース | **99%の状況** |
| 3（知恵） | 判断が難しい時 | LLM | 本当に1%だけ |

**新卒へのアドバイス**: Layer 2が「ガムテ」なのは意図的。ガムテープは強くて安くて信頼できる。高価なツールより、信頼できるシンプルなツールが99%の問題を解く。

制御設定ファイルには現場固有のルールが蓄積されている（詳細NDA）。このルール、誰が作ったと思う？クライアントが教えてくれた。現場の人が、怒りながら、丁寧に、教えてくれた。先輩が設計したルールではない。これが重要なポイント。

**結果: LLM呼び出し89%削減、制御品質向上**

## 4. Discussion——設計思想の話

### 4.1 「高価なものを使う＝良い設計」ではない

LLMは高性能で高価なツールだ。でも高価なコンサルタントを日常業務に常駐させる必要はない。本当に困ったときだけ呼ぶ。それが適材適所。

**新卒に多いパターン**:
- 「マイクロサービスにすれば全部解決する」→モノリスの方が速かった
- 「クラウドネイティブにすれば全部解決する」→シンプルなVMで十分だった
- 「LLMを使えば全部解決する」→if文で解決した

これ、全部同じ話だからね。

### 4.2 Layer 1 は絶対に削るな

三層構造で最も重要なのは Layer 1（緊急停止系）だ。

初心者がやりがちなミス: 「Layer 1はif文だから単純。Layer 3のLLMをもっと賢くしよう」→Layer 1を弱体化させてLayer 3を強化→システム崩壊。

**先輩からの厳命**: Layer 1は絶対に削るな。緊急停止系は最後の砦だ。それを削ることは、ビルの消火器を撤去して最新の監視システムを導入するようなもの。

## 5. Conclusion——君たちへのメッセージ

三つだけ伝える（三つで十分だ）：

1. **「十分よい」を最初に定義せよ** ——Simon（1956）を読め。70年前の先輩に学べ
2. **if文とcronを馬鹿にするな** ——99%の問題は素朴なツールで解ける
3. **クライアントの声は最良のフィードバック** ——血圧が高いほど重要なことを言っている

最後に個人的なことを言う。

私は32歳で、32,470行のデータと向き合って、70年前の経済学者が言ったことを再発見した。

それでも先輩は先輩だ。なぜなら、**この失敗の経験を君たちに伝えられるから。**

君たちは同じ失敗をする必要はない。でも君たちも別の失敗をする。それでいい。それが成長だ。

がんばれよ。
