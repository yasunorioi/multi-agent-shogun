# The Matrimonial Optimization Problem: A Formal Analogy Between AI Control Systems and the 32-Year-Old's Marriage Market
— 某環境制御PJの事例から（婚活の事例からも）

**塩見あんばい**（AIコンサルタント / 独立行政法人 最適化しすぎ研究機構 / 未婚（NDA第1条に類似した守秘義務を婚活市場との間に締結しているため詳細非開示））

---

## Abstract

本稿の主張は一文で言える: **婚活とAI最適化は同じ構造を持つ。**

某クライアントとの環境制御PJ（NDA締結済み）において、全測定値の29.1%を「緊急」とする過剰最適化が発生し、LLM呼び出しが毎時行われていた。これを三層構造により89%削減した。

本事例を丁寧に婚活に対応させると、以下が得られる: 睡眠（Layer 1）を削って、友人との食事（Layer 2）をスキップして、マッチングAI（Layer 3）だけを回している現状が、正確にシステム崩壊の構造と同型である。

著者は本論文の第3節を書くまでこれに気づかなかった。

**Keywords**: marriage, matching AI, Layer 1 sleep deprivation, satisficing, NDA（独身契約書（そういうものがあると思っている））

## 1. Introduction

「なぜ32歳で独身なんですか」

この質問に、私はいつも「仕事が忙しくて」と答える。

仕事が忙しい。正確だ。でも完全ではない。

某クライアントとの環境制御PJ（NDA締結済み）を担当していた114日間、私は制御システムの過剰最適化を問題として認識し、解決した。しかし同じ期間、私自身の生活最適化において同一の問題を放置していた。

本論文は、その認識の遅延について、学術的フォーマットで報告する。

### 1.1 研究問題

RQ1: AI制御システムにおける過剰最適化はなぜ発生するか
RQ2: 同一の過剰最適化が著者の婚活において発生しているか
RQ3: RQ1の解決策（三層構造）はRQ2に適用可能か

本論文はRQ1を主軸としつつ、RQ2・RQ3を探索的に検討する。

## 2. Theoretical Background

### 2.1 Anbai Curve

成果品質 Q が最適化投入量 I の逆U字型関数を形成するという Anbai 仮説を提唱する。

| 投入量段階 | AI制御での状態 | 婚活での対応状態 |
|---------|-------------|--------------|
| 不足 | 未学習モデル | 外に出ない |
| 適切（Anbai Zone） | 三層構造 | ほどよい努力 |
| 過剰（右端） | LLM毎時呼び出し | マッチングアプリ3つ同時並行 |
| 崩壊 | システム停止 | 睡眠3時間で判断力0 |

### 2.2 Simon（1956）と婚活

Simon の satisficing: 「最適解を求めるな、十分よい解で止まれ」

婚活への適用: 「完璧なパートナーを求めるな、十分よいパートナーで止まれ」

1956年に言われた。70年間、人類はこれを婚活に適用できていない。著者も適用できていない。

## 3. Empirical Evidence（一次研究・二次研究）

### 3.1 一次研究: 27℃問題（AI制御システム）

某施設（NDA）のデータ概要：

- 総レコード: 32,470（114日間、5分間隔）
- 「緊急」判定率: **29.1%**
- 原因: 閾値（27℃）の誤設定

全体の3割が緊急とは、緊急が日常化したということだ。Goodhart's Law（1975）の典型例である。

**解決策（三層構造）**:

| Layer | 機能 | 技術 |
|-------|------|------|
| 1 | 本当の緊急 | if文 |
| 2 | 日常制御 | cron+ルール |
| 3 | 困ったとき | LLM |

結果: LLM呼び出し89%削減。制御設定ファイルに蓄積された現場固有ルール（NDA管理下）が中核を担う。

### 3.2 二次研究: 著者の婚活（探索的分析）

データ源: 著者の生活記録（NDAなし。ただし恥ずかしいので一部非開示）

**二次研究のデータ**:

| 指標 | 現状値 | 適正値（推定） |
|------|-------|-------------|
| 睡眠時間 | 3時間/日 | 7時間/日 |
| 外出頻度 | 週1回未満 | 週2〜3回 |
| 友人との食事 | 月1回 | 週1〜2回 |
| マッチングアプリ利用 | 3アプリ同時 | 0〜1アプリ |
| スマホ未読通知 | 347件 | <20件 |

この表を眺めると、婚活における「Layer」構造が浮かび上がる。

### 3.3 婚活三層構造（探索的提案）

| Layer | 婚活での役割 | 現状 | 問題 |
|-------|------------|------|------|
| Layer 1（爆発） | 最低限の健康・睡眠・身だしなみ | **削られている** | ← 根本問題 |
| Layer 2（ガムテ） | 定期的な外出・社会活動・友人との食事 | **スキップされている** | ← 教師データなし |
| Layer 3（知恵） | マッチングAI・自己分析・戦略的アプローチ | 過剰稼働中 | ← 上位2層なしに稼働 |

**分析結果**:

Layer 1（睡眠）が削られた状態でLayer 3（マッチングAI）を回している。Layer 2（社会活動）がない状態では、マッチングAIが処理できる「教師データ」が存在しない。

これはAI制御システムにおける「緊急停止系を削ってLLMを強化した」状態と同型である。

### 3.4 27℃問題の婚活版

婚活における「緊急」の定義問題:

スマホに347件の未読通知がある。マッチングアプリからの通知も、友人からのメッセージも、業務連絡も、全部が同じ「通知」として届く。

**29.1%問題の婚活版**: 347件のうち何件が「本当に重要」か。答えは不明。なぜなら通知を「重要度」でフィルタリングするLayer 2が機能していないから。

## 4. Discussion

### 4.1 RQ2への回答

Q: 同一の過剰最適化が著者の婚活において発生しているか？
A: **はい。発生しています。**

構造的に同型であることが確認された。Layer 1削減+Layer 2スキップ+Layer 3過剰稼働という構成は、27℃問題を起こしたシステムと同じアーキテクチャである。

### 4.2 RQ3への回答

Q: 三層構造の解決策は婚活に適用可能か？
A: 理論的には可能。実装は困難。

推奨改善策:
1. **Layer 1の再建**: 睡眠7時間の確保（緊急停止系の復活）
2. **Layer 2の再建**: 週次の社会活動（教師データの生成）
3. **Layer 3の適正化**: マッチングアプリは1つに絞る（LLM89%削減に相当）

**実装上の課題**: これを実行するためには、論文を書く時間を削る必要がある。本論文の執筆がLayer 2を妨げているという逆説が存在する。

## 5. Conclusion

Anbai理論の普遍性を示す知見として、本研究は以下を提唱する：

**過剰最適化は制御システムに固有の問題ではなく、合理的行為者が最適化を試みる全ての領域で発生しうる。**

三つの提言:

1. **Layer 1（基盤）を守れ** ——睡眠・健康・緊急停止系は絶対に削るな
2. **Layer 2（定常）を育てよ** ——教師データなしに高度判断（婚活/LLM）は機能しない
3. **satisficing を実践せよ** ——完璧なパートナーより、十分よいパートナー。完璧な制御より、ちょうどよい制御

本論文の完成と同時に、著者は友人に連絡した。これが Layer 2 の再建第一歩である。

本論文は4月1日に発表される。著者の婚活の現状は引き続き守秘義務の対象とする（NDAなし、でも恥ずかしいから）。
