## 経緯（話すと長いんだけど聞いて）

某施設（NDA。地名も種別も言えない。でも存在する。本当に。信じて。）で114日間、32,470レコードのデータを取った。

最初の設計がこれ。

```python
# 初期コード（今思うと謎）
if temperature > 27:
    llm_response = call_llm("緊急！なんとかして！")
```

「温度が27℃より高かったら緊急として LLM に連絡する」。

……あのね、この設計のどこが問題か分かります？

**29.1%の時間が「緊急」になった**のよ。

温度の分布：

| 温度 | 割合 |
|------|------|
| 低め（<16℃） | 4.5% |
| 普通（16-23℃） | 36.3% |
| ちょっと高め（23-27℃） | 28.2% |
| **高め（>27℃）** | **29.1%** |

3割が緊急。

なんで3割が緊急になるのって話ですよ。緊急じゃないじゃん。日常じゃん。3割が緊急って、もう緊急の定義が壊れてる。

Goodhart's Law ってやつで（1975年、Charles Goodhartさんが言った）、「測定値が目標になった瞬間ダメになる」って話があって。これがそれ。27℃を緊急の基準にした瞬間、27℃は「緊急を正確に示す基準」じゃなくなった。

……眠いけど伝わってるかな。

## 解決策：三層構造という名の救世主

問題は何かというと、「LLMに何でもやらせようとしてた」こと。

解決策は「役割分担」。

```
Layer 1: if文（本当にやばいことだけ）
Layer 2: cron + ルールベース（日常の99%）
Layer 3: LLM（本当に困った時だけ）
```

コードで書くとこんな感じ：

```python
class AnbaiSystem:
    def __init__(self):
        self.rules = load_rules()  # 現場由来のルール（NDA管理下）
        self.llm = LLMClient()

    def process(self, data):
        # Layer 1: 絶対条件（削るな）
        if data.is_critical():
            emergency_alert()
            return "EMERGENCY"

        # Layer 2: ルールで99%解決
        result = self.rules.evaluate(data)
        if result.confident:
            return result.action

        # Layer 3: LLMは最終手段
        return self.llm.decide(data)
```

制御ルールはNDA管理下なんで詳細は書けないけど、クライアントからのフィードバックで積み上がったやつです。現場の知恵。AIが考えたルールじゃなくて、人間が怒りながら教えてくれたルール。大事にしてる。

## 教訓（眠い頭でまとめる）

3つだけ：

**① LLMは万能じゃない（当たり前だが忘れがち）**

高性能なモデルを使えばいいシステムができる、は嘘。適切な場所で適切な技術を使うことが大事。if文で解決できることはif文で解決する。

**② 閾値を決める前にデータの分布を見る**

29.1%問題は防げた。データの分布を確認して閾値を設定していれば、こうはならなかった。

**③ クライアントの声が最良のトレーニングデータ**

クライアントが怒ってるとき、そこにルールの芽がある。フィードバックをルールに変換する仕組みを作れ。
