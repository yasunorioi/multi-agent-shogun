# 新卒エンジニアへ：LLMを毎回呼ぶのをやめなさい（89%削減した話）

**はじめに**

最近、新卒・若手エンジニアと話していて気になることがある。

「LLMを使えばAI開発できる」「GPT-4oに全部やらせればいい」

**違います。**

某クライアントとの環境制御PJ（NDA締結済み）で、私はLLMを毎時呼んでいた。結果、全体の29.1%が「緊急」として処理されるシステムを作ってしまった。

この失敗と、そこから学んだことを、後輩エンジニアに伝える。

2. なぜ起きたか（原因の分析）

## 1. 何が起きたか

### データと状況

```
プロジェクト: 某クライアントとの環境制御PJ
NDA: 締結済み（施設詳細は非開示）
データ: 32,470レコード（114日間、5分間隔）
```

### 初期コード（問題あり）

```python
# これが間違いだった
def monitor(sensor_data):
    if sensor_data['temp'] > 27.0:
        # 毎時、LLMに判断を委ねる
        response = llm.chat("温度が高いです。どうすれば？")
        return execute(response)
    return "normal"
```

### 実際の結果

全データの温度分布を確認したら：

```
< 16℃  :  4.5%（1,461件）
16〜23℃: 36.3%（11,787件）
23〜27℃: 28.2%（9,157件）
> 27℃  : 29.1%（9,448件）  ← 全部「緊急」扱い
```

**問題**: 全体の29.1%が「緊急」として処理された。

新卒の君へ: **全体の3割が緊急状態というシステムは、正常に機能していない。**

---

## 2. なぜ起きたか（原因）

原因は二つある。

### 原因① Goodhart's Lawへの無知

Goodhart（1975）が言っている：「測定値が目標になった瞬間、それは良い測定値でなくなる」

「27℃を緊急の基準にする」と決めた瞬間、27℃は「緊急を正確に示す指標」として機能しなくなった。

**新卒へのアドバイス**: 閾値を決める前に、その値がデータ分布の中でどこにあるかを確認せよ。分布の29.1%が「異常」なら、その閾値は適切でない。

### 原因② 「新しいものを使う＝良い設計」という誤解

LLMが使えるから、毎回使おう。これが間違いだった。

適切なツールを適切な場所で使うことが良い設計。LLMは万能ではない。

## 3. どう直したか（三層アーキテクチャ）

### 設計思想の転換

**Before**: すべてLLMに聞く
**After**: 役割に応じて使い分ける

### 三層の役割

| Layer | 名前 | 技術 | 対象 |
|-------|------|------|------|
| 1 | 緊急停止 | if文 + 即時通知 | 絶対的な異常（数%) |
| 2 | 定常制御 | cron + ルールベース | **日常の99%** |
| 3 | 知的判断 | LLM | 本当に難しい1% |

### 実装

```python
class ThreeLayerController:
    """
    三層アーキテクチャの実装例。
    具体的なルールはNDA管理下のため省略。
    """

    def __init__(self):
        self.rule_engine = RuleEngine()  # 現場由来のルール
        self.llm = LLMClient()

    def process(self, data: SensorData) -> Action:
        # Layer 1: 絶対条件。削るな。
        if self._is_absolutely_critical(data):
            self.emergency_stop()
            self.notify_immediately()
            return Action.EMERGENCY

        # Layer 2: ルールで解決できるか？（99%はここで終わる）
        rule_result = self.rule_engine.evaluate(data)
        if rule_result.is_confident:
            return rule_result.action

        # Layer 3: LLMへ（本当に1%だけ）
        llm_result = self.llm.analyze(data)
        return llm_result.action

    def _is_absolutely_critical(self, data) -> bool:
        # if文。シンプルに書け。
        # 複雑な条件を書きたくなっても我慢せよ。
        return data.has_hard_limit_violation()
```

### Layer 2のルールについて

制御設定ファイルには現場固有のルールが蓄積されている（詳細はNDA管理下）。

これらのルールは、クライアントのフィードバックから生まれた。AIが設計したルールではない。

**新卒へのアドバイス**: クライアントが怒っている時、そこにルールの芽がある。フィードバックをきちんとルールとして文書化せよ。

### 結果

```
LLM呼び出し削減率: 89%
（毎時1回 → 約0.11回/時）
```

## 4. 君たちに伝えたいこと

### 先輩から3つだけ

**① if文とcronを馬鹿にするな**

```python
# これは「古い」コードではない。これは「正しい」コードだ。
if temp > HARD_LIMIT:
    emergency()
```

シンプルなif文は、理解しやすく、デバッグしやすく、信頼できる。Layer 1はif文で書け。

**② 閾値設定前にデータを見ろ**

```python
import numpy as np

# 閾値を決める前に、必ず分布を確認せよ
data = load_sensor_data()
print(f"25percentile: {np.percentile(data, 25):.1f}")
print(f"75percentile: {np.percentile(data, 75):.1f}")
print(f"95percentile: {np.percentile(data, 95):.1f}")
# → 95パーセンタイルが27℃なら、27℃を閾値にしたら5%が「緊急」
# → 70パーセンタイルが27℃なら、27℃を閾値にしたら30%が「緊急」（問題！）
```

**③ 「十分よい」を定義してから最適化せよ**

Herbert Simon（1956）は「satisficing」を提唱した。「最適解ではなく、十分よい解で止まれ」。1978年にノーベル経済学賞。

まず「何があれば十分か」を定義する。そこに到達したら止まる。止まることも設計の一部だ。

## まとめ

| 悪い設計 | 良い設計 |
|--------|---------|
| LLMに全部任せる | 役割に応じて使い分ける |
| 閾値を直感で決める | データ分布を確認してから決める |
| 「もっと賢く」を目指す | 「ちょうどよく」で止まる |

**LLMは強力だから、使いすぎてはいけない。**

高価な専門家を日常業務に常駐させるのは無駄だ。本当に困ったときだけ呼べばいい。

君たちが将来、29.1%問題を起こさないために、先輩はこの記事を書いた。

参考にしてくれたら嬉しい。でも失敗しても大丈夫。失敗から学んで、後輩に伝えることが仕事だ。
