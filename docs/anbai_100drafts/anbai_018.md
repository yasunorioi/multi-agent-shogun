## 問題1：LLM設計の話（技術パート）

某クライアントとの環境制御PJ（NDA締結済み。詳細は守秘義務のため非開示）。

```
データ: 32,470レコード（114日間、5分間隔）
施設: 某施設（NDA第2条：詳細非開示）
問題: 温度>27℃でLLMを呼び出す設計
結果: 全体の29.1%が「緊急」判定
```

**解析:**

```
<16℃:   4.5%（正常）
16-23℃: 36.3%（正常）
23-27℃: 28.2%（正常）
>27℃:   29.1% ← LLM毎回呼び出し（問題）
```

3割が緊急ならそれは日常だ。Goodhart's Lawの教科書例。

## 問題2：婚活の話（個人パート）

私のスマホには347件の未読通知がある。

```
LINE: 89件
マッチングアプリA: 112件
マッチングアプリB: 87件
マッチングアプリC: 43件
業務連絡: 16件
```

全部が「重要そう」に見える。全部が「見なければ」と思わせる。

でも本当に重要なものは何件か。

…分からない。347件を全部確認しないと分からない。でも347件は多すぎて確認できない。確認できないから未読が増える。増えるから全部が重要に見える。

**これ、29.1%問題と同じ構造だ。**

---

## 解決策：三層構造（技術と婚活の対応表）

### 技術的解決策

```python
# 三層アーキテクチャ
def process(data):
    # Layer 1: 本当の緊急だけ
    if data.is_absolutely_critical():
        return emergency_stop()

    # Layer 2: ルールで99%処理
    if result := rule_engine.check(data):
        return result

    # Layer 3: LLMは1%だけ
    return llm.analyze(data)
```

制御設定ファイルには現場固有のルール（NDA管理下）。クライアントの強いフィードバックから生まれたルール。血圧の高さに比例して正確。

**結果: LLM89%削減。品質向上。**

### 婚活への応用

| Layer | 技術 | 婚活 | 現状 |
|-------|------|------|------|
| Layer 1 | 緊急停止（if文） | 基本的健康・睡眠 | **削れている**🔴 |
| Layer 2 | 定常制御（cron） | 定期的な外出・食事・友人との交流 | **スキップ**🔴 |
| Layer 3 | 知的判断（LLM） | マッチングアプリ・戦略的活動 | **過剰稼働**🔴 |

**問題**: Layer 1（睡眠3時間）を削ってLayer 3（マッチングアプリ3つ）を過剰稼働させている。Layer 2（社会活動）がない状態では、Layer 3が処理すべき「教師データ」が存在しない。

**改善案（理論的）**:

```yaml
# 婚活システム改善設計
layer_1:
  sleep_hours: 7  # 現状: 3 → 目標: 7
  basic_health: true

layer_2:
  frequency: weekly  # 現状: monthly → 目標: weekly
  activities:
    - "友人との食事"
    - "外出"
    - "社会活動"

layer_3:
  apps: 1  # 現状: 3 → 目標: 1（89%削減に相当）
  strategy: "Layer 2で得た教師データを元に使用"
```

## まとめ

技術的まとめ:
- 過剰なLLM呼び出しは三層構造で解決できる
- Layer 1（if文）→ Layer 2（ルール）→ Layer 3（LLM）の順で処理せよ
- 89%削減は数字として正直だ

- Layer 1（睡眠）を守れ。他の全ては二次的だ
