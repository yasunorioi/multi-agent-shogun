# 新卒AIエンジニアへ贈るAnbai理論——なぜあなたのAIは99%サボっていいのか

**塩見あんばい**（AIコンサルタント / 独立行政法人 最適化しすぎ研究機構）
発表番号: P-037 | セッション: 次世代エンジニアへの知識継承
*本発表は、現場で同じ失敗を繰り返す新卒エンジニアへの警告として構成されている。著者自身の失敗記録を含む。*

---

## 【背景 / Background】

毎年4月、新たなAIエンジニアが業界に参入する。彼らは優秀だ。学習速度も高い。しかし、著者が複数の現場で観察した共通パターンがある。

**新卒エンジニアの典型的失敗パターン:**

```
Step 1: 要件定義段階で「最高精度のAIを作ろう」と宣言
Step 2: GPT-4相当モデルを全クエリに適用
Step 3: コストが予算の10倍になる
Step 4: 上司に怒られる
Step 5: if文で解決できることに気づく（遅い）
```

著者は某施設との環境制御PJで同様のパターンを経験した。本発表はその経験から導いたAnbai理論を、次世代エンジニアに伝えることを目的とする。

---

## 【Anbai理論の基礎 / Fundamentals】

### 定義（新卒向けに平易に）

**Anbai理論**: 「ちょうどいい精度で止まれ」という単純な話である。

ただしこれを「理論」と呼ぶ理由がある。単純な話であるにも関わらず、誰もそこで止まれないからだ。

## 【実証データ / Evidence（某施設、NDA管理下）】

### Case 1: 27℃問題——閾値設定の失敗

**状況:** 著者が参画した某施設の環境制御PJにて

| 項目 | 数値 |
|------|------|
| データ期間 | 2025年6〜9月、114日間 |
| 総レコード数 | 32,470件（5分間隔） |
| 緊急閾値 | 27℃ |
| 緊急判定された割合 | **29.1%（9,448件）** |

**何が起きたか:** 施設内温度が27℃を超えた瞬間をすべて「緊急」としてLLMに判断させようとした。結果、全データの3割が緊急扱いになった。

**後輩へのメッセージ:**
「全データの3割が緊急なら、それは緊急ではない」——この文章を10回書け。新卒エンジニアが最初に学ぶべきことはアーキテクチャではなく、この概念的真実だ。

これはGoodhart's Lawである。*"When a measure becomes a target, it ceases to be a good measure."* ——Goodhart（1975）。50年前に言われている。我々は毎年同じ失敗をしている。

### Case 2: 三層構造——解決策

失敗の後、著者はアーキテクチャを三層構造に変えた。

| 層 | 名称 | 手法 | 担当割合 |
|----|------|------|---------|
| Layer 1 | 爆発 | if文 + 緊急通知 | 本当の緊急のみ |
| Layer 2 | ガムテ | cron + ルールベース | **通常の99%** |
| Layer 3 | 知恵 | LLM | 本当に必要な時だけ |

**結果: LLM呼び出し89%削減。同時に制御品質は向上。**

**後輩へのメッセージ:**
Layer 2の「ガムテ」という名前に注目してほしい。ガムテープで貼り付けた応急処置のようだが、それが99%を担う。エンジニアとしての美意識は理解できる。しかし現場はガムテで回っている。それを恥じるな。誇れ。

## 【学術的基盤 / Theoretical Grounding】

### Herbert Simon「満足化（Satisficing）」——1956年

Simonは人間の意思決定について「最適解ではなく、十分に良い解で止まる」合理性を提唱した。1978年ノーベル経済学賞。

**後輩へのメッセージ:**
70年前の経済学者が言ったことを、君たちはAPIコールとして支払っている。Simonを読もう。コードを書く前に。

### Goodhart's Law——1975年

指標を目標にした瞬間、指標は機能しなくなる。27℃問題はその実例だ。

**後輩へのメッセージ:**
精度を上げれば良いという考え方そのものが、精度という指標をGoodhart's Lawの犠牲にする可能性がある。「精度を上げることは本当に目標か」と問え。

## 【よくある誤解と反論 / Common Misconceptions】

### 誤解1: 「if文は恥ずかしい」

**反論:** 某施設の制御品質を支えているのはif文とルールベースだ。LLMが判断する機会は全体の1%未満。その1%のために99%のインフラがある。恥じるものは何もない。

### 誤解2: 「精度は高ければ高いほど良い」

**反論:** 29.1%問題を見よ。精度閾値を厳しくするほど、緊急が増えた。「精度を上げる」ことが「より多くのデータを緊急と判定する」ことと等価だった。これは精度の向上ではなく、Goodhart's Lawの作動だ。

### 誤解3: 「LLMは汎用的に使える」

**反論:** LLMは万能だが、万能を使いすぎるとコストで死ぬ。エキスパートをすべての問いに答えさせるな。本当に難しい問いにだけ使え。

## 【実務への適用指針 / Practical Guidelines】

**後輩AIエンジニアへの5か条:**

1. **まずif文で解けないか考えよ** — 解けるなら解け。それがLayer 2だ
2. **閾値を設定する前にデータ分布を確認せよ** — 27℃問題を繰り返すな
3. **LLMを呼ぶ条件を厳しく設定せよ** — 呼べばいいと思った瞬間が過剰最適化の始まりだ
4. **Goodhart's Lawを常に疑え** — 改善したい指標が本当に目的と連動しているか
5. **Anbai Pointで止まれ** — それがプロの仕事だ

## 【結論 / Conclusions】

新卒AIエンジニアへ。

あなたは賢い。技術も学んでいる。しかし技術の適用タイミングこそが現場では最も重要だ。

某施設の事例は示す: **89%のLLM呼び出しは不要だった。** それは失敗だったが、修正できた。修正後の方が良くなった。

Herbert Simonは1956年に言った。「最適解を追求することは、必ずしも合理的ではない」と。

あなたが新卒として現場に入る前に、この言葉を一度噛み締めてほしい。

**「ちょうどいい」はいつも、「最高」より難しい。**
