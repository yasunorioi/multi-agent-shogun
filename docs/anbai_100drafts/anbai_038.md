## 【理論的枠組み / Framework】

### 三層構造モデル（Figure 1）

某施設の環境制御システムは以下の三層で動いている:

```
Layer 3: 知恵層（LLM）     ← 難しい判断
Layer 2: ガムテ層（ルール） ← 日常の制御
Layer 1: 爆発層（if文）    ← 絶対的基盤
```

**この三層構造は、人間の基本的行動設計と同型である。**

| 制御システム | 婚活 | 機能 |
|-------------|------|------|
| if文 + 緊急通知 | 睡眠・食事・最低限の健康管理 | Layer 1: 生存基盤 |
| cron + ルールベース | 定期的な外出・友人との交流・自然な出会い | Layer 2: 日常の積み重ね |
| LLM（条件付き呼び出し） | マッチングAI・自己分析ツール・恋愛心理学 | Layer 3: 高度な判断支援 |

## 【実証データ1: 某施設の失敗から学ぶ】

### 27℃問題——重要な前提

某施設では、温度センサーの緊急閾値を27℃と設定した結果、**全データの29.1%（n=9,448/32,470）が「緊急」と判定された**。

この問題の核心: **全体の3割が「緊急」なら、それは緊急ではなく日常である。**

これはGoodhart's Lawの作動である。指標が目標となった瞬間、指標は機能を失う。

**婚活への適用（Figure 2）:**

| 某施設の指標 | 婚活の指標 | 機能不全のパターン |
|-------------|-----------|-----------------|
| 27℃ = 緊急 | 「気になる人」= 緊急対応 | 全員が「気になる人」になる |
| 29.1%が緊急判定 | 全プロフィールが高評価になる | いいねの99%が無意味になる |
| Goodhart's Law作動 | マッチング指標の形骸化 | スワイプ疲れ |

### 三層構造による解決——某施設の場合

LLMを毎回呼ぶのではなく、Layer 2（ルールベース）で99%を処理し、本当に必要な時だけLayer 3（LLM）を呼ぶ。

**結果: LLM呼び出し89%削減。制御品質は向上した。**

## 【実証データ2: 著者の婚活失敗から学ぶ（n=1）】

### 診断（Figure 3）: 著者の現在の婚活アーキテクチャ

```
著者の現在の婚活アーキテクチャ（問題あり）:

Layer 3（過剰稼働）: マッチングAI日次使用、自己分析ツール週次、
                     恋愛心理学書籍月次10冊、ホロスコープ日次確認
Layer 2（停止中）: 友人との食事 → 最後は3ヶ月前
                   自然な出会いの場への参加 → 不定期（稀）
Layer 1（削減中）: 睡眠時間 → 1日3時間に最適化済み
```

**診断結果: Layer 1を削ってLayer 3を稼働させている**

これは某施設が27℃問題を修正する前のアーキテクチャと同型である。Layer 2をスキップしてLayer 3に直行する設計は、某施設においても著者の婚活においても、同様に機能しない。

### 教師なし学習の問題（Figure 4）

婚活における根本的問題は「教師データの欠如」である。

```
機械学習の文脈で言えば:

分類タスク: 「この人がいい/よくない」を学習する（教師あり学習）
要件: 実際のデートという教師データが必要

著者の現状:
- Layer 3（マッチングAI）を高頻度で呼んでいる
- しかし実際のデート（教師データ）が発生していない
- 観測データのないモデルを学習しようとしている

これは教師なし学習である。
教師なし学習でできることはクラスタリング（似た者同士のグループ化）のみ。
分類（この人〇/×）には教師データが必要。
```

著者は現在、クラスタリングを婚活だと思っている。

## 【Anbai Curveの婚活への適用 / Application to Dating】

### Figure 5: 婚活版Anbai Curve

```
婚活成果
  ↑
  │           ╭── Anbai Zone ──╮
  │         ╱                   ╲
  │       ╱    「ちょうどいい行動量」 ╲
  │     ╱          Layer 2          ╲___  ← 著者の現在地
  │╱ Layer 1基盤なし
  └──────────────────────────────→ 婚活への最適化投入量
          Anbai Point（Layer 2適切稼働）
```

**Key insight:** 婚活成果を最大化しようとしてLayer 3（最適化ツール）を最大稼働させることは、Layer 1（基本的な健康・睡眠）とLayer 2（自然な人間関係）を犠牲にしながら過最適化している状態である。

### Anbai Pointの定義（婚活版）

- Layer 1: 睡眠7時間、食事3食、基本的健康管理
- Layer 2: 月に2〜3回、友人や新しい人との食事・交流
- Layer 3: 上記2層が安定している時に、補助的に使用

著者の現在の配分はこの逆である。

## 【考察 / Discussion】

### なぜ婚活は過最適化されるのか

婚活は「成果が見えにくい」という特性を持つ。某施設の環境制御では「LLM呼び出し回数」「制御品質スコア」という定量的指標があった。婚活では指標が曖昧だ。

指標が曖昧だからこそ、人は「なんとなく頑張っている感」のあるLayer 3（マッチングアプリ使用回数、プロフィール改善回数）を過剰稼働させる。

これは某施設における27℃問題と同構造だ。「緊急を多く検知すること」が「制御品質向上」と混同されたように、「婚活ツールを多く使うこと」が「婚活の質向上」と混同されている。

### Goodhart's Lawの婚活版

「マッチング数を婚活成功の指標とした瞬間、マッチング数は婚活成功を示す指標として機能しなくなる」——これがGoodhart's Lawの婚活版である。

著者には現在、マッチング数を増やすことに特化した行動パターンが観察されている。デートには至っていない。

## 【結論 / Conclusions】

1. **婚活の構造はAI環境制御と同型である**（三層モデルが適用可能）
2. **Layer 1を削減してLayer 3を稼働させることは機能しない**（実証済み、n=1）
3. **教師データなき学習はクラスタリングに終わる**（教師データ=実際の交流）
4. **Anbai Pointは「Layer 2の適切稼働」にある**（ツールより人との接触）
