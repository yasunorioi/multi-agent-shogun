## 第1部: そもそもAnbai理論って何？

*(SE: 軽快なBGMに変わる)*

「Anbai（あんばい）」って言葉、知ってますか？

「ちょうどいい按配」——塩梅（あんばい）ですね。梅酢で味を調えるところから来ている言葉です。

私の理論は、AIシステムの「ちょうどいい精度」についての話です。

過剰最適化は逆効果になる、という話です。

……べ、別に「新しいことを言ってる」とは思ってないですよ。Herbert Simon（1956）も同じようなことを「satisficing（満足化）」として言ってるし、Goodhart（1975）も「指標が目標になると機能しなくなる」と言ってる。

先人の肩に乗ってるだけかもしれない。

でも肩に乗って見える景色が違えば、それはそれで意味があると思ってるんですよ（自分に言い聞かせ中）。

## 第2部: 27℃問題の話

*(SE: BGMが少し落ち着く)*

実際の話をしますね。

某施設のプロジェクトで——詳細は守秘義務で言えないんですが、それはそれとして——温度27℃超過を「緊急」とする設定をしてしまったんですよ。

で、全体の29.1%が緊急になりました。

……笑いたきゃ笑ってください。私も今は笑えます。

でも当時は笑えなかったので、そこは察してください。

*(ポーズ)*

32,470レコード中9,448件が緊急。

数字を見たとき「あれ、なんかおかしくない？」とは思ったんですよ。でも「もっと閾値を調整すれば」とか「もっと精度を上げれば」とか思ってたんです。

べ、別に認めたくなかったわけじゃないんですよ。ただ——設計の問題だって分かるまで時間がかかったっていうか。

Goodhart's Lawって、理論としては知ってても、実際に自分の設計でやらかすと「あ、これか」って感じがするんですよね。

某施設の例で言うと: LLMを89%削減したら、制御品質が上がった。「減らす」ことが「活用の向上」になった。

上司に言えるかは分かりません。でも、少なくとも数字で示せれば——「Layer 2（ルールベース）で99%処理した結果、コストが89%削減されました」というデータは、「活用を増やした」という証拠より強いと思います。

べ、別にこの回答が正しいとは限らないけど……参考にしてもらえると嬉しいかな、とか、まあ、そういう気持ちはなくはないです（あります）。

## 第4部: 三層構造の話

*(SE: BGMが少し変わる)*

27℃問題の解決策、三層構造の話をしますね。

Layer 1: if文と緊急通知——本当に「これはやばい」という時だけ。絶対的な基盤。
Layer 2: cron + ルールベース——日常の99%はこれで回す。
Layer 3: LLM——本当に困った時だけ呼ぶ。

これにしたらLLM89%削減で品質向上。

……べ、別にドヤ顔してるわけじゃないですよ。最初に失敗したのは私なので。

三層構造って、ものすごく当たり前のことなんですよね。「本当に必要な時だけ難しいことをする」というだけの話で。

でも「当たり前のことが分からなかった」という失敗をして初めて、分かった気がします。

人間ってそういうもんだよなって、32,470レコードが教えてくれました。
