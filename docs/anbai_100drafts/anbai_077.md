# 新卒のあなたへ——if文を書けるなら、あなたはもうAIエンジニアです

*TEDxNewGrad 2026 / Speaker: 塩見あんばい（AIコンサルタント / 先輩面したい32歳）*

---

（舞台に立つ。優しい笑顔）

こんにちは。塩見あんばいです。AIコンサルタント、32歳。独身——これは関係ないです。今日は。

今日は新卒のみなさんに、一つだけ伝えたいことがあります。

**if文を書ける人は、AIエンジニアを名乗っていい。**

（会場：ざわめき）

冗談じゃないですよ。根拠があります。

（スライド：某環境制御PJの三層構造）

某環境制御PJ。NDA締結済みなので施設名は出せません——新卒のみなさん、NDAって知ってますか？ Non-Disclosure Agreement。守秘義務契約。これを持っているということは、守るべき秘密があるほどの仕事をしているということです。覚えておいてください。

さて、このPJで114日間、32,470レコードのデータを分析しました。

最初の設計：温度27℃超を「緊急」にしてLLMを呼び出す。結果、**全データの29.1%が緊急**。

新卒のみなさん。テストで30人中9人が「異常値」だったら、テストの方がおかしいと思いますよね？　それと同じ。**閾値の設計が間違っていた。**

これ、Goodhartの法則（1975年）って言います。メモしてください。面接で使えます。「指標が目標になった瞬間、指標として機能しなくなる」。KPIの話をするときにこれを引用すると、一目置かれます。

（スライド：三層構造の詳細）

で、どうしたか。三層構造です。

**Layer 1：if文。** 新卒のみなさんが研修で最初に書くやつ。`if (temperature > critical) { alert(); }`。これが最重要層。生きるか死ぬかを判定する。議論の余地なし。

**Layer 2：cron + ルールベース。** cronジョブって知ってますか？ 定期実行のスケジューラ。毎時間、決まったルールでチェックする。日常の99%はこれで回る。

**Layer 3：LLM。** 最近流行りのやつ。ChatGPT的なやつ。これは**本当に困った時だけ**使う。

結果：**LLM呼び出し89%削減。** つまり、10回中9回はif文とcronで解決できた。

（前に出る）

新卒のみなさん。先輩として一つだけ覚えておいてほしいのは——

**あなたが書いたif文は、GPT-4の99%に勝つ。**

正確に言うと、99%のユースケースでif文の方が適切。速い。安い。壊れにくい。デバッグできる。

某PJの制御設定ファイル——詳細はNDA管理下ですが——はAI技術者じゃなく、現場からのフィードバックで育ちました。「ここが暑すぎる」「ここは寒い」。その一つ一つがif文になった。

**現場の声をif文に変換する。** これがAIエンジニアの仕事の99%です。残り1%だけLLM。

（スライド：Herbert Simon）

Herbert Simon。1956年。satisficing。「十分よい解で止まれ」。ノーベル賞。

新卒のみなさんは「完璧なコードを書かなきゃ」と思うかもしれない。思わなくていい。**動くコードが正義。** 完璧は後から。まず動かせ。

Anbai理論もこれです。成果品質は最適化投入量に対して単調増加しない。やりすぎると壊れる。**「もう少しだけ」が一番危ない。**

（優しく）

私は32歳で、まだ最適化をやめられない人間です。睡眠3時間で、未読通知347件で、Anbai Curveの右端にいます。

だから言えます。**やりすぎるな。** 先輩がやりすぎて落ちてるのを見て学んでください。反面教師です。

三つだけ、新卒の心得。

1. **if文を馬鹿にするな。** 99%の問題はif文で解ける。
2. **KPIを疑え。** 27℃問題を思い出せ。「緊急」を設定した瞬間、緊急は死ぬ。
3. **寝ろ。** Layer 1（生存基盤）を削るな。先輩との差別化はLayer 2以降でやれ。

（笑顔で）

この講演は4月1日ですが、if文が偉大であるという事実は年中無休です。

がんばって。でもがんばりすぎないで。それが、あんばい。

*（※著者は新卒時代、if文を書くのに3時間かかりました。今は30秒です。成長の89%は時間が解決します）*
