## 下ごしらえ

1. **温度データを114日間寝かせる。** 5分間隔で32,470レコードになるまで発酵させる。某施設で（施設名はNDA管理下）。

2. **緊急閾値27℃を設定する。** この時点では正しい判断に見える。味見するとちょうどいい塩加減のはず。

3. **結果を観察する。** 全データの29.1%が「緊急」と判定される。29.1%。ここで異変に気づくこと。全体の3割が緊急なら、それは**塩を入れすぎた状態**。

> **シェフのメモ:** 塩分濃度0.8%が最適（Anbai Zone）。2.0%でしょっぱい。5.0%で食べられない。27℃閾値は塩分5.0%に相当する。Goodhart（1975）が予言した味の崩壊。

## 調理手順

### Step 1: Layer 1を仕込む（強火・短時間）

```
材料: if文 1行
火加減: 強火（即座に判定）
```

if文を一行書く。`if 致死的 then 停止`。これが全ての基盤。味噌汁で言えば出汁。出汁がなければ何を入れても味が決まらない。

**調理のコツ:** if文は複雑にしないこと。条件分岐を3つ以上入れたくなったらLayer 2に移す。if文は単純であるほど美味しい。

---

### Step 2: Layer 2を煮込む（弱火・じっくり）

```
材料: cronジョブ 適量、ルールベース 各種
火加減: 弱火（毎時間コトコト）
```

cronで毎時間チェック。ルールベースで日常の99%を処理する。弱火でじっくり煮込む工程。時間がかかるが、ここが味の8割を決める。

某PJの制御設定ファイル——NDA管理下だが調理法は開示可能——はこのLayer 2にあたる。現場のフィードバック（クライアントの怒り）で味が整った。**血圧に比例する品質の出汁。**

> **シェフのメモ:** 秘伝のタレは足し算でしか育たない。引き算（ルール削除）は怖くてできない。何が入っているかわからないが、味は確実に向上している。これは科学ではなく発酵。

---

### Step 3: Layer 3を仕上げに少々（最後に一振り）

```
材料: LLM API 少々
火加減: 強火で一瞬（1時間予報のみ）
```

LLMは仕上げの一振り。トリュフオイルのようなもの。入れすぎると全てがトリュフの味になる。

**入れすぎの指標:** LLM呼び出しが全体の10%を超えたら入れすぎ。某PJでは89%削減して最適になった。つまり元々90%以上が不要だった。

---

### Step 4: 味見（Anbai判定）

Simon（1956年）のsatisficing基準で味見する。**「完璧な味」ではなく「十分よい味」で止める。**

もう少し塩を入れたくなる。もう少しLLMを呼びたくなる。**その衝動を抑えることが、Anbai理論の核心。**

> **⚠️ 注意:** 味見のたびに調味料を足す人がいる。Goodhartの法則により、味見自体が味を変える。味見は最小限に。
