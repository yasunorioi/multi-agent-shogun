# 『味見しすぎた料理人の告白』——Anbai Curveの右端で焦がしたレシピ

*塩見あんばいの最適化キッチン vol.6 〜反省献立編〜*
*調理時間: 114日+気づくまでの全時間 / 難易度: ★（最初からこうすればよかった） / カロリー: 削減すべきだった分*

---

## このレシピを書く理由

私は料理を焦がしました。

最適化という名の火を強くしすぎた。味見を繰り返しすぎた。調味料を入れすぎた。結果、Anbai Curveの右端——Over-Optimization Zone——に到達した。

このレシピは「正しい作り方」ではなく、**「間違えた作り方」の記録**です。

## 間違えた調理手順（反面教師として）

### 間違い1: 火が強すぎた

27℃を緊急にした。某PJ。114日間。32,470レコード。NDA管理下。

火を強くすれば早く煮えると思った。緊急閾値を低くすれば安全だと思った。結果、全データの3割が緊急。鍋の3割が焦げた状態。

**正しくは:** 火加減を三段階に分ける。Layer 1は強火で一瞬（if文）。Layer 2は弱火でじっくり（cron）。Layer 3は火を止めてから余熱で（LLM・必要時のみ）。

### 間違い2: 調味料を入れすぎた

LLMを全ケースで呼んだ。トリュフオイルを全料理にかけた。

89%削減して気づいた。**ほとんどの料理に出汁（if文）と塩（ルールベース）だけで十分だった。** トリュフが必要なのは本当に特別な一皿だけ。

### 間違い3: 味見しすぎた

Goodhartの法則。味見のたびに「もう少し」を足す。足すたびに基準が変わる。永遠に「ちょうどいい」に到達しない。

某PJの制御設定ファイル。NDA管理下だが構造は言える。あれは**味見しないで育てた**。現場のフィードバック（怒り）をそのままルールにした。分析せず、最適化せず、ただ蓄積した。秘伝のタレ。

**味見しなかったから、ちょうどいい味になった。** 逆説。

## 正しいレシピ（反省を踏まえて）

### 材料（修正版）

| 材料 | 分量 | 反省前 → 反省後 |
|------|------|----------------|
| if文 | 1行 | 変更なし（最初から正解だった） |
| cron | 適量 | 不足 → 適量（99%カバー） |
| LLM | 少々 | 過剰 → 少々（89%削減） |
| 睡眠 | 7時間 | 3時間 → 回復中 |

### 手順（修正版）

1. **まずif文を書く。** これだけで料理の骨格が決まる。
2. **cronで煮込む。** 急がない。弱火。時間をかける。
3. **LLMは最後の最後。** 味が決まらないときだけ。決まってるなら不要。
4. **味見は2回まで。** 3回目以降はGoodhartが発動する。

## 覚醒の瞬間

某PJの最終報告書を書いていたとき。深夜3時。三層構造の図を描きながらカレーを温めていた。

図には「Layer 1を削ってLayer 3を強化するな」と書いた。

カレーを食べながらふと気づいた。**私は今、睡眠（Layer 1）を削って分析（Layer 3）をしている。** 図に描いたアンチパターンを、リアルタイムで実行している。

Simon（1956年）。satisficing。十分よい解で止まれ。

私はカレーを食べ終わった時点で止まるべきだった。報告書は朝でよかった。分析は明日でよかった。**十分よい状態はもう来ていた。** 通り過ぎただけ。
