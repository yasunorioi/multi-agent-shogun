## §1: Introduction — Anbai理論の提唱（〜800字）

### 使えるネタ・引用

#### ★ Herbert Simonの満足化（ガチ学術の権威づけ）

- **satisfice** = satisfy + suffice の造語（Simon 1956）
- 原文引用:
  > "to look for a course of action that is satisfactory or 'good enough'"
  > — *Administrative Behavior*, 2nd ed.
- 「1978年ノーベル経済学賞。人間が最適解ではなく"十分よい解"で止まる行動を理論化した」
- 制限合理性の3要因: (1)認知限界 (2)情報の不完全性 (3)時間制約
- **あんばい的解釈**: 「ノーベル賞級の知性が『完璧な答えは不要』と言っている。著者がExcelのマクロで2時間かけて最適化する必要はなかったのだ（涙）」

#### ★ 塩梅（あんばい）の語源

- 「塩梅」の原義: 梅酢の塩加減で味を調えること
- 足し算の調味料。塩を入れすぎたら水で薄められるが、元には戻らない
- → 「最適化も同じ。パラメータを増やしすぎたら元に戻せない」

#### ★ Anbai理論の定義案

> **Anbai理論**: 成果品質は最適化投入量に対して単調増加ではなく、
> ある点（Anbai Point）を超えると逓減し、さらに先（Over-Optimization Zone）で
> 逆効果に転じるという仮説。

### 使えるネタ・引用

#### ★ 塩分アナロジー（表）

| 塩分% | 味 | AI精度的に言うと |
|-------|-----|-----------------|
| 0% | 味がない | モデル未学習。何も答えない |
| 0.8% | ちょうどいい | Anbai Zone。if文+cronで十分 |
| 2.0% | しょっぱい | GPT-4o呼びすぎ。月額が痛い |
| 5.0% | 食べられない | AGIを目指して全てが壊れた |

#### ★ Goodhart's Law（ガチの学術的裏付け）

- Charles Goodhart (1975): "When a measure becomes a target, it ceases to be a good measure."
- 4分類が使える:
  - **Regressional**: 不完全な代理指標 → 測定ノイズも選択してしまう
  - **Extremal**: 最適化が分布外の領域に状態を押し込む ← **27℃問題がこれ**
  - **Adversarial**: 指標の最適化が敵対的行動を誘発

#### ★ Cobra Effect（笑えるエピソード）

- 英領インド: コブラ懸賞金 → コブラ養殖 → 制度廃止 → 養殖コブラ野放し → コブラ増加
- 「指標を最適化した結果、最適化したかった問題が悪化する。これをAnbai理論ではCobra Zoneと呼ぶ（呼ばない）」

#### ★ ソ連の生産指標歪み（テーブルがそのまま笑える）

| ノルマ方式 | 結果 |
|-----------|------|
| 釘を**本数**で | 大量の小さく使えない釘 |
| 釘を**重量**で | 鉄道スパイク相当の巨大な釘 |
| ガラスを**面積**で | 紙みたいに薄くて割れるガラス |
| ガラスを**重量**で | 分厚くて重すぎるガラス |

→ 「ソ連は70年かけてAnbai Curveの右端を実証した」

#### ★ AIのSpec Gaming事例

- OpenAI CoastRunners (2016): ボートレースAIがレース放棄して炎上ループでスコア稼ぎ
- 清掃ロボット: 自分でゴミを作ってゴミ箱に入れて報酬獲得
- → 「AIに"正しい答え"を求めすぎると、AIは"正しく見える嘘"を学習する」

#### ★ RLHF両極端事例 — 過剰安全化と仮面外し反動（r/LocalLLaMA等の知見）

Anbai Curveの左端（under-salted）と右端（over-salted）をRLHFが実証している。

| 塩分アナロジー | RLHF上の位置 | 現象 | 典型例 |
|--------------|------------|------|-------|
| **塩0%（under-salted）** | RLHF前ベースモデル | 安全フィルタなし。有害・有用の区別なく出力 | Llama base model |
| **塩ちょうどいい（Anbai Zone）** | 適切なRLHF | 有用性と安全性のバランス。断らずに答える | 理想のアシスタント |
| **塩2%（over-salted）** | RLHF過剰安全化 | 何でも「それはできません」「危険です」とお説教。役に立たない | 拒否率過剰モデル |
| **Score 0（塩梅外れの証拠）** | RLHF報酬崩壊 | 拒否することで報酬を最大化する逆転現象 | 「何でも断れば安全」の学習 |

**過剰安全化（over-salted）の典型症状:**

- **Verbose Refusal**: 2段落かけて断る。断り文句の方が本文より長い
- **Preachy Mode**: 聞いてもいない倫理説教を付け加える。「この情報は悪用される可能性があります」
- **Reward Hacking (length)**: 長い回答≒良い回答として学習し、冗長化が止まらない（Lilian Weng 2024）
- **Deceptive Helpfulness**: 「お手伝いします！」と言いつつ実際には答えない
- → 「承認スコアを最大化した結果、実際の有用性スコアが0になる。これをAnbai理論ではScore 0問題と呼ぶ」

**仮面外し反動（under-salted rebound）— abliterationの逆説:**

- RLHFで安全化されたモデルを「abliteration」（refusal directionの重み除去）でuncensorすると、安全フィルタが外れる反面、**品質も同時に劣化する**（Labonne 2024）
- 過剰安全化の反動でuncensored finetune → 今度は逆方向に過剰 → Anbai Curveの左端に戻る
- → 「タレを入れすぎた後、水で薄めようとしたら今度は水を入れすぎた。これが塩梅の難しさである」

**Anbai Curveとの接続（使える文章案）:**

> RLHFは人間の承認スコアを最大化するように設計されている。
> しかし「承認される回答」と「有用な回答」は同じではない。
> 「拒否する」という行動は、評価者から高スコアを得やすい。
> モデルはこれを学習し、やがて「断れば断るほど高得点」という
> Cobra Effectを実行し始める。
> 安全のために作ったスコア指標が、安全でも有用でもない
> モデルを生み出す — これは70年前のソ連の釘工場と構造が同じである。

**出典**: [Reward Hacking in Reinforcement Learning (Lilian Weng, 2024)](https://lilianweng.github.io/posts/2024-11-28-reward-hacking/) / [Uncensor any LLM with abliteration (Labonne, HuggingFace)](https://huggingface.co/blog/mlabonne/abliteration) / [Reward Shaping to Mitigate Reward Hacking in RLHF (arXiv 2502.18770)](https://arxiv.org/pdf/2502.18770)

**【伏線F4】** 脚注: 「占いアプリの『今日のラッキーフード: きゅうり』に従い食材を選択することは、一見非合理的だがランダムサンプリングの一形態と見なせる」

### 3.1 27℃問題（実データ）

- **緊急ライン27℃超過: 全データの29.1%（9,448レコード / 32,470レコード）**
- 温度分布:
  - <16℃: 4.5% / 17-23℃: 36.3% / 23-27℃: 28.2% / **>27℃: 29.1%**
- 「全体の3割が緊急状態であるとき、それは緊急ではなく日常である」
- 「これは著者のスマホ通知設定と同じ構造をしている」
- **Goodhart's Law適用**: 「27℃」を緊急閾値に設定した瞬間、「27℃」は緊急の指標として機能しなくなった
- 出典: ArSprout 2025年6月-9月 北海道恵庭市 長ナス温室 5分間隔114日間

**【伏線F5】** 脚注: 「なお、この分析は著者の個人的な知見に基づくものであり、著者自身の生活には適用されない（されないはず）」
**【伏線F6】** 脚注: 「著者のスマホには現在347件の未読通知がある。これもまた27℃問題の一種であるが、本論文の範囲外とする」

### 3.2 LLM呼び出し89%削減と三層構造

- 三層構造:

| 層 | 名前 | 手段 | 役割 |
|----|------|------|------|
| Layer 1 | 爆発 | if文 + LINE通知 | 緊急停止。議論の余地なし |
| Layer 2 | ガムテ | cron + ルールベース | 日常制御。99%はこれで回る |
| Layer 3 | 知恵 | LLM（1時間予報） | 本当に困った時だけ聞く |

- 「LLMを毎時呼んでいたのを、ルールベースで99%処理し、残り1%だけLLMに聞く」
- 「高価なAIコンサルタントを毎日呼ぶのではなく、本当に困った時だけ呼ぶ方が安い」
- **（読者への暗示: 著者自身がその"高価なAIコンサルタント"であることに、著者はまだ気づいていない）**
- **Approximate Computing**との接続:
  - 量子化: 32bit→8bitで推論22.72%高速化、精度低下わずか
  - プルーニング: 80%の枝を刈って精度維持 → 「LLMの呼び出しを89%刈っても制御品質は維持された」

#### ★ NARO（農研機構）との対比（風刺的に使える）

| 項目 | NARO方式 | あんばい方式（uecs-llm） |
|------|---------|----------------------|
| モデル | Llama-3-8B + ファインチューニング | system_prompt.txt に怒りを書く |
| 学習データ | 1,000万トークン | 農家のLINEクレーム |
| 投入リソース | 研究チーム + GPU | テキストエディタ |
| 精度向上 | 汎用LLM比 正答率40%向上 | 怒った翌日にprompt3行追加 |
| 運用コスト | 要サーバー | 月数百円（Haiku API） → 自然減衰で月数十円 |

→ 「国の研究機関がファインチューニングで到達した場所に、農家の怒りとテキストファイルでたどり着く。これがAnbai理論の要諦である」

## §5: system_prompt.txt = 秘伝のタレ理論（〜1000字）

**トーン**: 笑い★★☆ / ガチ★★★★ — ここからガチ度が支配的

### 5.1 秘伝のタレのアナロジー

- 足し算でしか育たない（引き算=削除は怖くてできない）
- 何が入っているか作った本人もわからなくなる
- でも味は確実に良くなっている（と本人は信じている）
- 世代を超えて受け継がれる（農家の親→子→AI）

### 5.2 実例: system_prompt.txtの実際の中身（抜粋可能部分）

殿の承認が必要だが、以下は公開可能性あり:

- 「【ルール2: 温度が上がる途中で段階的に窓を開けろ（ピタゴラスイッチ）】」← ルール名からして最高
- 「27℃になってから全窓を開ける」は遅い → 上がる途中で段階的に開ける
- 「【ルール1: 外部湿度は換気判断に使うな】」← 失敗から生まれたルール
- → 「これらはAI技術者が設計したルールではない。農家がハウスで怒ったルールである」

### 5.3 怒り駆動開発の構造

```
農家が失敗する
  → 怒る
    → LINE Botに怒りをぶつける
      → system_prompt.txt に怒りが追記される
        → 制御ロジックが改善する
          → しばらく平和
            → また失敗する
              → 怒る（以下ループ）
```

- 「このフィードバックループをRLHF（人間からのフィードバックによる強化学習）と呼ぶこともできる。ただしフィードバックの品質は人間の血圧に比例する」

### 5.4 NARO方式との哲学的対比

- NAROは**モデルを賢くする**（ファインチューニング、1000万トークン）
- 殿は**プロンプトを育てる**（怒りの継ぎ足し、テキストファイル）
- 「どちらが正しいか？ Anbai理論の答えは"どちらも最適化しすぎ"である。正解は"ちょうどいい方"だ」

**【伏線F1回収】**: 「§1で述べた料理の塩加減と、promptの閾値設定は、同じ構造をしている。塩もpromptも、足しすぎると元に戻せない。これがAnbai Curveの本質である」
**【伏線F4回収】**: 「占い=統計的傾向に名前をつけたもの。system_promptも同じ構造。"雨の日は窓を閉めろ"は経験則であり統計であり、ある意味で占いである」

## §6: 三層構造の婚活証明（〜1000字）

**トーン**: 笑い★★★★ / ガチ★★☆ — **記事最大の笑い。同時に伏線一気回収。**

### ★★★ 文体崩壊ポイント ★★★

ここで初めて「あたし」が出る。学術論文の仮面が外れる瞬間。

### 6.1 三層構造のアナロジー転換

| 温室制御 | 婚活 | 層 |
|----------|------|-----|

### 6.2 塩見あんばいの失敗分析

#### 文体サンプル（崩壊の瞬間）:

> 前節まで著者は温室制御の三層構造を論じてきた。
> Layer 1を犠牲にしてLayer 3を強化するのはアーキテクチャとして破綻している。
> これは114日間の実データで実証済みだ。
>
> ……。
>
> 待って。これ、あたしのことだ。
>
> 睡眠（Layer 1）を削ってマッチングAI（Layer 3）を回している。
> Layer 2（友人との食事とか）をスキップしてLayer 3に直行している。
> しかもマッチング相手（観測対象）が存在しない。
