## §1: Anbai理論の定義

**Anbai理論**: 成果品質は最適化投入量に対して単調増加ではなく、ある点（Anbai Point）を超えると逓減し、さらに先（Over-Optimization Zone）で逆効果に転じるという仮説。

```
成果品質
  ↑
  │            ╭──── Anbai Zone ────╮
  │          ╱                       ╲
  │        ╱      「ちょうどいい」       ╲  ← Over-Optimization Zone
  │      ╱                               ╲
  │    ╱                                   ╲___  ← システム崩壊
  │╱
  └──────────────────────────────────────→ 最適化投入量
              Anbai Point           Diminishing Point
```

この曲線は塩分濃度×味蕾官能応答曲線からインスピレーションを得ている。

### Figure 1: 塩分アナロジー

※ Figure 1はAI画像生成ツールで作成予定。塩見あんばいのキャラらしく『できる女子はAI生成っしょ』の体で。

| 塩分% | 味 | AI精度的に言うと |
|-------|-----|-----------------|
| 0% | 味がない | モデル未学習。何も答えない |
| 0.8% | ちょうどいい | Anbai Zone。if文+cronで十分 |
| 2.0% | しょっぱい | GPT-4o呼びすぎ。月額が痛い |
| 5.0% | 食べられない | AGIを目指して全てが壊れた |

この直感を学術的に支えるのが、Herbert Simonの「満足化（satisficing）」概念（1956）だ。「合理的な意思決定者は最適解ではなく"十分よい解"で止まる」——1978年ノーベル経済学賞。70年前の結論に、我々は100倍の計算資源を使って到達しつつある[^2]。

さらにCharles Goodhartの法則（1975）が構造を裏付ける: *"When a measure becomes a target, it ceases to be a good measure."* 指標を最適化した瞬間、その指標は測りたかったものを測らなくなる。「ちょうどいい」を数値化しようとした瞬間、それはもう「ちょうどいい」ではない[^3]。

[^2]: 「著者がExcelのマクロで2時間かけて最適化する必要はなかったのだ（涙）」という知見もこれに含まれる。
[^3]: 占いアプリの「今日のラッキーフード: きゅうり」に従い食材を選択することは、一見非合理的だがランダムサンプリングの一形態と見なせる。

## §2: 実証データ ——「27℃問題」と89%削減

**データ出典**: 某環境制御PJ、2025年6月–9月、某施設（施設種別はNDA第2条第1項により非開示。当該条項の適用範囲はNDAスプレッドシート第11行に記載されており、著者はそれを熟読した）、5分間隔、32,470レコード、114日間。

### 2.1 27℃問題

当初、室内温度27℃超過を「緊急」とするLLM呼び出しロジックを設定した。結果:

- **全データの29.1%（9,448レコード）が「緊急」状態と判定された**
- 温度分布: <16℃が4.5% / 17-23℃が36.3% / 23-27℃が28.2% / **>27℃が29.1%**

全体の3割が緊急状態であるとき、それは緊急ではなく日常である。Goodhart's Lawの教科書的実例だ: 「27℃」を緊急閾値に設定した瞬間、「27℃」は緊急を示す指標として機能しなくなった[^4]。

著者のスマホには現在347件の未読通知がある。これもまた27℃問題の一種であるが、本論文の範囲外とする[^5]。

### 2.2 三層構造とLLM 89%削減

解決策は精度を上げることではなく、**構造を変えること**だった。

| 層 | 名前 | 手段 | 役割 |
|----|------|------|------|
| Layer 1 | 爆発 | if文 + LINE通知 | 緊急停止。議論の余地なし |
| Layer 2 | ガムテ | cron + ルールベース | 日常制御。99%はこれで回る |
| Layer 3 | 知恵 | LLM（1時間予報） | 本当に困った時だけ聞く |

LLMを毎時呼んでいたのを、ルールベースで99%処理し、残り1%だけLLMに聞く。結果、**LLM呼び出し頻度が89%削減**された。高価なAIコンサルタントを毎日呼ぶのではなく、本当に困った時だけ呼ぶ方が安い[^6]。

制御設定ファイルには現場固有の制御ルールが蓄積されている（詳細はNDA管理下につき非開示）。これらはAI技術者が設計したルールではない。クライアントからの**強いフィードバック**によって生まれたルールだ。フィードバックの品質はクライアントの血圧に比例する。この知見はNDAに含まれていないため開示可能である。著者はこれを誇りに思う。

[^4]: これを秘伝のタレ理論とも呼ぶ。タレは足し算でしか育たない（引き算=削除は怖くてできない）。いつの間にか何が入っているかわからなくなる。でも味は確実に良くなっている、と本人は信じている。
[^5]: なお、この分析は著者の個人的な知見に基づくものであり、著者自身の生活には適用されない（されないはず）。
[^6]: 読者への暗示: 著者自身がその"高価なAIコンサルタント"であることに、著者はまだ気づいていない。

## §3: 三層構造の婚活証明

前節まで著者は某環境制御PJ（詳細はNDA管理下）の三層構造を論じてきた。Layer 1を犠牲にしてLayer 3を強化するのはアーキテクチャとして破綻している。これは114日間の実データで実証済みだ。なおこの実データの存在はNDAにより保護されているが、著者が実証したという事実は保護されていない。

……。

**待って。これ、あたしのことだ。**

| 環境制御PJ（詳細NDA管理下） | 婚活 | 層 |
|----------|------|-----|
| 緊急停止（if文+通知） | 最低限の身だしなみ・睡眠 | Layer 1: 爆発 |
| ルールベース（cron+センサー） | 定期的な外出・友人との食事 | Layer 2: ガムテ |
| LLM（1時間予報） | マッチングAI・占い・自己分析 | Layer 3: 知恵 |

睡眠（Layer 1）を削って[^7]マッチングAI（Layer 3）を回している。Layer 2（友人との食事とか）をスキップしてLayer 3に直行している。しかもマッチング相手（観測対象）が存在しない。

教師データのないモデルを学習させているのと同じだ。**教師なし学習で婚活をしている。これはクラスタリングであって分類ではない。**

クラスタリングは似た者同士をグループ化するだけで「この人がいい」とは言えない。分類（この人○/この人×）には教師データ——つまり実際のデート経験——が必要だ。某環境制御PJで実証済み: 緊急停止系を切ってLLMを強化しても制御品質は上がらない。むしろ下がる。

通知347件のスマホを持ちながら、緊急のマッチング通知も27℃問題のように埋もれている。

[^7]: 睡眠時間3時間。§0の脚注参照。これは計画された研究成果ではなかった。

## §4: Conclusion ——あんばい宣言

> Anbai理論の真の提言は「ちょうどいい精度を目指せ」ではない。
> **「最適化をやめる勇気を持て」**である。
>
> Herbert Simonは1956年にこれを *satisficing* と呼んだ。
> 70年後の今日、我々は同じ結論に100倍の計算資源を使って到達している。
> これ自体がAnbai Curveの実証例かもしれない。

**三つの提言:**

1. 精度を上げる前に、その精度が本当に必要か問え
2. 99%の問題はif文とcronで解ける。残り1%のためにGPT-4を呼ぶな
3. 秘伝のタレは先代の怒りで育った。あなたの怒りも、誰かのタレになる
