## 1. なぜ2段階に分離するか

### 現状の問題

現行の監査フロー（STEP 0〜4.6）は直線的に進むため、**仕様不適合の成果物に対しても品質チェック（ルーブリック採点）のコストが発生する**。

具体的には：
- 指定ファイルが存在しない
- コミットハッシュが架空（ハルシネーション）
- 「完了」宣言なのに検証コマンドを実行していない（"should pass" 宣言）

…といった **自明なFAIL** を早期に検出できず、5観点×3点の詳細採点まで走ってしまう。

### 2段階分離のメリット

```
[現行]  STEP 1 → 2 → 3 → 3.5 → 4（ルーブリック）→ 5 → 6
                                        ↑
                              仕様不適合でもここまで進む

[新方式] Phase 1（仕様準拠）→ FAIL即終了
                ↓ PASS
         Phase 2（品質）→ ルーブリック採点
```

| 効果 | 説明 |
|------|------|
| **コスト削減** | Phase 1 FAIL の場合、ルーブリック採点スキップ。監査コスト最大50%削減 |
| **差し戻し高速化** | 「ファイルが存在しない」ならルーブリックを見ずに即差し戻し |
| **明確なFAIL理由** | Phase 1 FAILは「仕様不適合」、Phase 2 FAILは「品質不足」と区別できる |
| **ハルシネーション防止** | 既存STEP 3.5（コミット実在確認）をPhase 1の必須チェックとして明確化 |

## 2. Phase 1 — 仕様準拠チェック

**目的**: 成果物が「存在して・指示通りの内容で・本当に届いているか」を確認する。

**判定**: 1項目でも FAIL → Phase 1 FAIL（即終了。Phase 2スキップ）

### チェック項目

| # | チェック | 手段 | FAILの例 |
|---|---------|------|---------|
| P1-1 | subtask description と成果物の合致 | DB確認 + 報告書 Read | 指示と異なるファイルを修正 |
| P1-2 | detail_ref（高札report）の内容と成果物の合致 | curl detail_ref + Read | 報告書と実ファイルが乖離 |
| P1-3 | 指定ファイルの実在確認 | ls -la / Glob / Read | target_path のファイルが存在しない |
| P1-4 | コミット実在確認（STEP 3.5） | git ls-remote + git log | 報告コミットハッシュが不在 |
| P1-5 | verification-before-completion 確認 | 報告書の証跡確認 | "should pass" / "looks correct" のみで実行証跡なし |

### P1-5: verification-before-completion とは

足軽が「完了」を宣言する前に、**必ず検証コマンドを実行してその結果をエビデンスとして提出する**原則。

**禁止表現（NGワード）**:
- 「動くと思います」「should pass」「looks correct」「問題ないはずです」

**合格条件（エビデンスベース）**:
- テスト実行ログが報告に含まれている
- curl/httpx による実動作確認結果が記載されている
- `git log` / `ls -la` 等の実コマンド出力が報告に含まれている
- 「X行で確認」「出力: Y」のような具体的証拠がある

**お針子の確認手順**:
1. 足軽報告の body / findings を確認
2. 「完了」宣言に対応する実行エビデンス（コマンド出力・テスト結果等）が存在するか確認
3. エビデンスが「推測・予想」のみ → P1-5 FAIL

### Phase 1 FAIL 時の処理

```
Phase 1 FAIL → YAML: result=rejected_trivial（原則）
              → findings: ["[仕様準拠] P1-X FAIL: {理由}"]
              → send-keys で老中に通知
              → Phase 2 スキップ
```

FAIL理由が設計判断を要する場合（例：仕様書そのものが不明瞭）は `rejected_judgment` でも可。

## 3. Phase 2 — コード品質チェック

**対象**: Phase 1 を全項目 PASS した成果物のみ。

**方式**: 既存の5観点ルーブリック（15点満点）をそのまま活用。変更なし。

```
┌────────────────┬──────────┬──────────────┬──────────────┬──────────────┐
│ 観点           │ 0点      │ 1点          │ 2点          │ 3点          │
├────────────────┼──────────┼──────────────┼──────────────┼──────────────┤
│ 完全性         │50%未満   │50-70%        │70-90%        │90%以上       │
│ 正確性         │致命的誤り│重大な誤り    │軽微な誤り    │誤りなし      │
│ 書式           │3件以上   │2件違反       │1件違反       │違反なし      │
│ 一貫性         │重大不整合│部分的不整合  │軽微な不整合  │完全整合      │
│ ★横断一貫性   │矛盾     │部分差異      │概ね一致      │完全一致      │
└────────────────┴──────────┴──────────────┴──────────────┴──────────────┘
※ 横断一貫性は高札NG時はスキップ（12点満点で判定）
```

### 判定閾値（変更なし）

| スコア | 判定 | YAML result |
|-------|------|-------------|
| ≥12/15 | 合格 | approved |
| 8-11/15 | 条件付き合格 | conditional_approved |
| 5-7/15 | 要修正（自明） | rejected_trivial |
| ≤4/15 | 要修正（判断必要） | rejected_judgment |

## 4. verification-before-completion の監査チェックリスト

お針子が Phase 1 で確認する「足軽がやるべき事前検証」の一覧。

### 足軽が完了宣言前に行うべき検証（お針子がPhase 1で確認する）

| タスク種別 | 必須検証 | エビデンス形式 |
|-----------|---------|--------------|
| ファイル修正 | `Read` で修正内容を読み直す | 「行XXで確認：{内容}」を報告に記載 |
| コード実装 | テスト実行 or 動作確認コマンド | テスト出力 or curl結果を報告に記載 |
| git commit/push | `git log` / `git ls-remote` で実在確認 | コミットハッシュ + ショートlog |
| ドキュメント作成 | 作成ファイルを `Read` で読み直す | 「{パス}のX行を確認」を記載 |
| API設定変更 | curl による疎通確認 | HTTPステータスコードと応答本文 |

### NGワード検出（お針子の自動FAILトリガー）

以下のフレーズが足軽報告の「完了根拠」として使われていれば P1-5 FAIL：

- 「〜のはずです」「should work」「should pass」
- 「問題ないと思います」「動くと思います」「looks correct」
- 「確認していませんが」「未テストですが」
