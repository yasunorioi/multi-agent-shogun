# ADR-0001: YAML+DB二層通信プロトコル

- **Status**: accepted
- **Date**: 2026-02-08

## Context

初期のshogunシステムでは、タスク割当・報告・進捗管理を全てYAMLファイルで行っていた。タスク数の増加に伴い以下の問題が発生した：

1. **YAML肥大化**: 報告YAMLが数百行に膨張し、Read時のトークン消費が膨大に
2. **データロスト**: YAMLの手動編集ミスで報告が消失するケースが発生
3. **検索不能**: 過去の完了タスクをYAMLから検索するのが困難
4. **権限管理不在**: 全エージェントがYAMLに自由に書き込み可能で、データ整合性が保証できない

## Decision

通信プロトコルv2として、YAML+DB二層構造を採用する。

- **Layer 3a (YAML通信)**: 進行中タスクの割当・報告のみ保持。揮発層
- **Layer 3b (没日録DB)**: 完了済みタスクをSQLiteに永続化。家老のみ書き込み可
- **没日録CLI**: `python3 scripts/botsunichiroku.py` でDB操作を集約
- **DB書き込み権限**: 家老（karo-roju）のみに限定。足軽・お針子はYAML経由で通信

代替案として検討されたもの：
- **全DB方式**: YAML廃止してDB直接操作 → Claude CodeのEdit toolでYAMLの方が操作しやすいため不採用
- **全YAML方式**: 現状維持 → 肥大化・検索性の問題が解決しないため不採用

## Consequences

- **メリット**: YAML肥大化の抑制（直近10件保持ルール）、過去タスクのSQL検索、権限分離によるデータ整合性
- **デメリット**: 家老のDB操作負荷増、CLI学習コスト、二重管理の複雑性
- **制約**: DB書き込みは家老のみ。足軽が直接DBに書くことは禁止（F001-F006の拡張）
- **将来**: 高札API（kousatsu）によるDB検索で、全エージェントが過去タスクを参照可能に（ADR-0005参照）
