## §1 EDINET DB 技術調査

### 1a. サービス概要

[EDINET DB](https://edinetdb.jp/) は Cabocia株式会社が提供する、金融庁EDINETの有価証券報告書XBRLデータを構造化・正規化して提供するSaaS。
金融庁EDINET APIのラッパーではなく、**独自のETLパイプライン**でXBRLを名寄せ・構造化した上で、REST API + MCPで提供する。

| 項目 | 内容 |
|------|------|
| 対象企業 | 約3,847社（全上場企業） |
| データ期間 | FY2020〜2025（6年分）、四半期はFY2014〜2023 |
| 会計基準 | JP GAAP 91.8% / IFRS 7.5% / US GAAP 0.7% |
| 更新頻度 | 財務データ 毎朝8:00 JST / 企業マスタ 毎月1日 |
| カバレッジ | 基本指標 99%+ / CF 99%+ / 詳細BS・PL 80%+ / R&D 50%未満 |
| 処理ロジック | 部分公開（標準element_idマッピング+REGEXPフォールバック公開、独自拡張ID全容は非公開） |
| 商用利用 | 全プランOK |

### 1e. データ構造: 84フィールド内訳

**損益計算書(P/L)**:
売上高、売上原価、売上総利益、販管費、営業利益、経常利益、特別利益、特別損失、税前利益、当期純利益、包括利益、減価償却費、研究開発費

**貸借対照表(B/S)**:
総資産、流動資産、固定資産、有形固定資産、無形固定資産、のれん、ソフトウェア、棚卸資産(9科目)、現預金、有利子負債(6科目計)、流動負債、固定負債、純資産、資本金、利益剰余金

**キャッシュフロー(CF)**:
営業CF、投資CF、財務CF、現金等残高、FCF(算出)、設備投資額

**投資指標**:
EPS、希薄化EPS、PER、BPS、ROE、ROA、自己資本比率

**配当・株式**:
1株配当、中間配当、配当性向、発行済株式数、配当利回り

**算出指標(自動計算)**:
EBITDA、DEレシオ、営業利益率、純利益率、粗利率、前年比成長率(各種)、3年CAGR(売上・利益・EPS・配当の4種)

**ランキング対応(19種)**:
ROE / 営業利益率 / 純利益率 / ROA / 自己資本比率 / 健全性スコア / PER / EPS / 配当利回り / 配当性向 / 売上高 / FCF / 売上成長率 / 純利益成長率 / EPS成長率 / 3年CAGR×4種

### 1g. 金融庁EDINET APIとの比較

| 項目 | 金融庁EDINET API | EDINET DB |
|------|-----------------|-----------|
| データ取得 | 日付ループ+ZIP展開+XBRL解析 | 1 API call |
| 名寄せ | 自前実装(200+パターン) | 84フィールド標準化済み |
| 会計基準 | JP/IFRS/US個別対応 | 統一カラム |
| レート制限 | 暗黙(3-5秒間隔) | 明示(100回/日Free) |
| 保守負荷 | 月数時間 | ゼロ |
| 月額 | ¥0(公共API) | ¥0(Free) |
| MCP連携 | なし | あり(22ツール) |
| テキスト | XBRL生データ | 有報全文テキスト |

**EDINET DBの付加価値**: XBRLの「element_id → 統一カラム名」の名寄せ処理が最大の価値。
IFRS企業の独自拡張ID処理（REGEXP フォールバック）で99.6%の企業の売上高取得を実現。

### 2b. 「棍棒で殴れる」指標の選定

EDINET DBの84フィールドから殿の哲学に合致する「棍棒で殴れる」指標を選定:

**Tier 1: 最も殴れる（物理・実物指標）** ★★★
| EDINET DB指標 | 棍棒理由 | daily_risk.py対応 |
|-------------|---------|-----------------|
| 有形固定資産 | 工場・設備=物理の動き | なし(新規) |
| 設備投資額 | CapEx=将来の物理投資 | なし(新規) |
| 棚卸資産 | 在庫=物理の滞留 | なし(新規) |
| 営業CF | 現金=嘘をつけない | なし(新規) |
| FCF | 自由現金=殴れる余裕 | なし(新規) |

**Tier 2: 殴れる（ファンダメンタルズ）** ★★
| EDINET DB指標 | 棍棒理由 | daily_risk.py対応 |
|-------------|---------|-----------------|
| 有利子負債 | 借金=測定可能なリスク | なし(新規) |
| DEレシオ | レバレッジ=定量リスク | なし(新規) |
| 自己資本比率 | 安全余裕 | なし(新規) |
| 配当性向 | 株主還元=経営意志 | なし(新規) |
| ROE | 資本効率 | なし(新規) |

**Tier 3: 殴れない（捨てる）** ★
| EDINET DB指標 | 理由 |
|-------------|------|
| PER | 株価連動=予測領域 |
| のれん | 会計上の概念=物理なし |
| 研究開発費 | カバレッジ50%未満で信頼性低 |

### 2c. daily_risk.pyとの統合設計

現行daily_risk.pyは**マクロ指標**（FRED/yfinance）→ リスクスコア → SAFE/CAUTION/DANGER。
EDINET DBは**ミクロ指標**（企業財務）を提供する。

**統合アーキテクチャ**:

```
┌─────────────────────────────────────────────────────────┐
│  daily_risk.py (既存: マクロリスク)                        │
│  FRED(7指標) + yfinance(3指標) + 月次FRED(4指標)          │
│  → SCORE_WEIGHTS → risk_label (SAFE/CAUTION/DANGER)      │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼  trigger: CAUTION or DANGER
┌─────────────────────────────────────────────────────────┐
│  edinetdb_drill.py (新規: ミクロ深掘り)                    │
│  EDINET DB API → 監視対象企業のCF・有利子負債・設備投資     │
│  → 企業別リスクフラグ (棍棒指標の急変検出)                  │
│  → agent-swarm finance板に投稿                            │
└─────────────────────────────────────────────────────────┘
```

**重要**: daily_risk.pyに直接EDINET DBを組み込まない。
理由: daily_risk.pyは920行で安定稼働中。別スクリプトとして分離し、
daily_risk.pyのDANGER/CAUTION判定を**トリガー**としてミクロ深掘りを発火させる。

### 2e. 大株主データの活用

`get_shareholders` / `search_shareholders` の活用設計:

```

  ↓ フィルタ: 保有目的「投資」かつ保有割合変動 > ±2%
  ↓
スマートマネーシグナル:
  - 機関投資家が「増やした」企業 → ポジティブシグナル
  - 機関投資家が「減らした」企業 → ネガティブシグナル

  「マクロ危機 AND 機関投資家が逃げている」= 強い警告
  「マクロ危機 BUT 機関投資家が買い増し」= 過剰反応の可能性
```

### 2f. 投資仮説との接続

殿の投資仮説: **アジアインフラ × GIS × オルタナティブデータ**

EDINET DBの接続点:
1. **アジアインフラ**: 建設・鉄鋼・セメント・重機企業の設備投資額+棚卸資産 → インフラ投資の先行指標
2. **GIS連携**: 企業の有形固定資産(工場・倉庫)の地理分布は有報テキストに記載 → `get_text_blocks`で抽出可能(将来)
3. **オルタナティブデータ**: 大量保有報告書 = 機関投資家の動きは典型的なalt data
4. **コモディティ連携**: daily_risk.pyのFRED_COMMODITY(アルミ・鉄鉱石)とEDINET DBの原材料コスト企業を突合

## §4 リスク・見落とし分析

### 4a. データ品質リスク

1. **EDINETソースデータの遅延**: 有報提出 → EDINET公開 → EDINET DB反映 の遅延は通常1営業日。
   決算短信(TDNet)はget_earningsで別途取得可能だが、速報性では証券会社APIに劣る。
   - **影響度**: 低。daily_risk.pyはマクロ指標がメインで、企業財務は補完的位置づけ。

2. **カバレッジの穴**: R&D費は50%未満。IFRS企業のフォーマット差異による欠損。
   - **影響度**: 中。Tier 1指標（CF、設備投資、棚卸資産）は99%+カバーで問題なし。

3. **会計基準間の比較性**: JP GAAP/IFRS/US GAAPの統一カラム化は「近似」。厳密な比較には注意。
   - **影響度**: 低。Lasso正則化は近似でも十分機能する（ノイズに強い）。

### 4b. API廃止リスク

4. **Cabocia社の継続性**: スタートアップ（2025年β、2026年正式リリース）。
   - **影響度**: 中。個人プロジェクトではなく法人運営。プレスリリース公開済み。
   - **対策**: 処理ロジックが部分公開されているため、万が一廃止されても自前実装の参考になる。
   - **フォールバック**: 金融庁EDINET API v2（永続）への切り替え。名寄せコードは自前実装が必要（推定200+行）。

5. **API仕様変更リスク**: MCPプロトコル `2025-03-26` 版。MCP仕様自体がまだ若い。
   - **対策**: REST APIを主系統、MCPを副系統にすることで依存を最小化。

### 4c. 競合データソース比較

| サービス | 月額 | 財務データ | 株価データ | MCP | 有報テキスト |
|---------|------|:--------:|:--------:|:---:|:----------:|
| **EDINET DB** | **¥0(Free)** | **◎** | ✗ | **◎** | **◎** |
| J-Quants API | ¥0(個人) | △ | ◎ | ✗ | ✗ |
| 金融庁EDINET | ¥0 | ◎(要自前処理) | ✗ | ✗ | ◎(XBRL生) |
| Bloomberg | $2,000+/月 | ◎ | ◎ | ✗ | ◎ |
| Yahoo Finance | ¥0 | △ | ◎ | ✗ | ✗ |

**結論**: 月額¥0で企業財務に特化するならEDINET DBが最適。株価はyfinance(既存)で補完。

### 4d. 見落としの可能性

6. **企業コード体系の整合**: EDINET DBはEDINETコード(E0XXXX)を使用。systradeの既存コード体系との変換が必要。
   - **対策**: search_companiesで証券コード→EDINETコードの変換は可能。

7. **四半期データの活用**: FY2014-2023の四半期データがあるが、API経由のアクセス方法が不明。
   - **対策**: get_financialsのパラメータ確認が必要（実装時に調査）。

8. **バックテスト期間**: FY2020-2025の6年はバックテストには短い。daily_risk.pyのバックテスト（2008リーマン、2020コロナ等）との整合が取れない。
   - **影響度**: 高。EDINET DBデータだけではバックテスト不十分。FRED時系列(数十年)との組み合わせが必須。
