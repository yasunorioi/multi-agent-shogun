> **Version**: 1.0.0
> **策定日**: 2026-03-16
> **subtask**: subtask_912 / cmd_418
> **由来**: 経産省 石油情報システム要件定義(2023年)のエッセンスをshogun規模に簡略化

## 1. 経産省フレームワークからの抽出

3124行の報告書から取り入れるべき骨格は3つだけ。

### 1-1. 流れ（As-Is → To-Be → 要件）

```
現状整理 → 課題抽出 → 対応方針 → 将来像 → 要件定義
 (As-Is)                              (To-Be)
```

経産省ではこの流れに数ヶ月かける。shogunでは**cmdの起案段階で1回だけ通す**。

### 1-2. 課題→施策の優先度マトリクス

経産省方式: 難易度(低/中/高) × 重要度(高/低) の6象限。

```
          難易度
        低    中    高
重要度
  高    P1    P2    P3    ← 短期的取組（即cmdに）
  低    P4    P5    P6    ← 中長期（棚上げ or 将来cmd）
```

shogun適用: **P1〜P3を「今のcmdで対応」、P4〜P6を「棚上げ」**とする。

### 1-3. 要件の3区分

| 区分 | 経産省での定義 | shogunでの読み替え |
|------|-------------|-----------------|
| **業務要件** | 誰が・何をできるべきか | **「何ができればOKか」** |
| **機能要件** | システムがどう動くべきか | **「どう動けばOKか」** |
| **非機能要件** | 性能・可用性・セキュリティ等 | **「壊れずに動くか」** |

### 捨てたもの（shogunに不要）

- BPMN業務フロー図（農家1人に業務フローは不要）
- ステークホルダーヒアリング・アンケート（殿が全ステークホルダー）
- クラウド化検討・認証方式検討（RPi + SQLite、認証は.env）
- ガバメントクラウド・ISMAP（個人農家に無関係）
- 移行計画・教育計画（焼き直し5分で終わる）

## 2. プロジェクト設計骨格テンプレート

新規プロジェクトまたは大きめのcmdを起案するとき、将軍/家老がこの骨格を埋める。

### 2-1. 1枚シート（必須）

```yaml
# --- プロジェクト設計骨格 ---
project: {project_id}
cmd_id: cmd_XXX
date: YYYY-MM-DD

# ① 現状（As-Is）: 今どうなっているか
as_is: |
  1行で現状を書く。

# ② 課題: 何が問題か
issues:
  - id: I-1
    description: "課題を1行で"
    importance: high  # high / low
    difficulty: low   # low / mid / high
    priority: P1      # P1(即対応) 〜 P6(棚上げ)

# ③ 将来像（To-Be）: どうなっていたいか
to_be: |
  1行で理想を書く。

# ④ 要件
requirements:
  business:   # 業務要件: 何ができればOKか
    - "ユーザーが〇〇できる"
  functional: # 機能要件: どう動けばOKか
    - "△△した時に□□が起きる"
  non_functional: # 非機能要件: 壊れずに動くか
    - "SDカード32GBに収まる"
    - "オフラインで動作する"
```

### 2-2. projects/*.yaml への追加フィールド案

既存の `projects/*.yaml` に以下を任意追加:

```yaml
# 既存フィールドはそのまま
name: ...
status: ...
priority: ...

# 追加（任意）
requirements:
  business:
    - "農家がLINEで開けろ/閉めろできる"
    - "灌水量と生育の相関が見える"
  functional:
    - "5分毎にセンサー+画像ペアを保存"
    - "閾値超過時にLINE通知"
  non_functional:
    - "RPi SD 32GBに収まる"
    - "回線切れても制御継続"
    - "Picoは壊れたら焼き直し5分"
```

## 3. 完了判定基準の枠組み

### 3-1. 「動いたらdone」→「要件を満たしてdone」

| 判定レベル | 基準 | 使用場面 |
|-----------|------|---------|
| **L1: 動いた** | コードがエラーなく動作する | 調査・ドキュメント系subtask |
| **L2: 機能要件OK** | 指定された動作をする | 実装系subtask（標準） |
| **L3: 全要件OK** | 業務+機能+非機能すべて満たす | リリース判定・Wave最終subtask |

### 3-2. subtask完了時の確認フロー

```
足軽: タスク完了
  │
  ▼ L1チェック: 動くか？（テスト実行 or curl確認）
  │   → NO → 修正
  │
  ▼ L2チェック: 機能要件を満たすか？
  │   → cmd/subtaskのdescription内の機能要件と照合
  │   → NO → 修正
  │
  ▼ L3チェック: 非機能要件を満たすか？（リリース時のみ）
  │   → ストレージ・応答時間・オフライン動作等
  │   → NO → 報告に「非機能要件未達: XXX」を明記
  │
  ▼ 報告（done / done with caveats）
```

### 3-3. お針子監査との連携

既存の2段階レビュー（docs/review_two_stage.md）に要件チェックを組み込む:

| Phase | 既存チェック | 追加チェック |
|-------|------------|------------|
| **Phase 1（仕様準拠）** | P1-1〜P1-5（ファイル実在、コミット実在等） | **P1-6: 機能要件との照合**（cmdのrequirements.functionalと成果物の合致） |
| **Phase 2（品質）** | 5観点ルーブリック（15点満点） | 非機能要件は**Phase 2の「完全性」観点**に含める（新観点の追加は不要） |

P1-6の詳細:
- subtask descriptionまたはcmd設計骨格にrequirements.functionalがある場合のみ実施
- 各機能要件に対し「成果物が要件を満たしているか」を確認
- 1つでも未達 → Phase 1 FAIL（ただし部分完了の場合は足軽報告のcaveatsを確認）

## 4. cmd化の判断基準

課題一覧からcmdを起こす際の優先度変換。

### 4-1. 優先度マトリクスの使い方

```
1. 課題を列挙（issues配列）
2. 各課題に importance(high/low) と difficulty(low/mid/high) を付与
3. 優先度Pを決定:
   - high重要度 + low難易度  = P1（今すぐcmd化）
   - high重要度 + mid難易度  = P2（次のcmdで対応）
   - high重要度 + high難易度 = P3（設計してからcmd化）
   - low重要度               = P4-P6（棚上げ）
4. P1-P3をcmd化、P4-P6は没日録にメモだけ残す
```

### 4-2. difficulty（難易度）の判断基準

| 難易度 | shogunでの意味 |
|--------|--------------|
| **low** | 既存コード修正のみ。設計不要。1 subtask で完了 |
| **mid** | 新規ファイル追加 or 複数ファイル変更。設計書が必要。2-5 subtask |
| **high** | アーキテクチャ変更 or 外部連携。軍師に戦略立案を依頼。5+ subtask |

### 4-3. importance（重要度）の判断基準

経産省の3観点をshogunに読み替え:

| 観点 | 経産省 | shogun |
|------|--------|--------|
| **継続性** | 既存業務を維持できるか | 既存の動作を壊さないか |
| **安定性** | 業務品質への影響 | データが壊れないか・誤制御しないか |
| **効率性** | 非効率な業務の改善 | 殿の手間が減るか・API代が減るか |

1つでも「高」に該当 → importance: high

## 5. docs/skill_format_v1.md への影響

影響なし。スキルは「繰り返し使える作業パターン」であり、プロジェクト要件とは別レイヤー。

ただし、スキルのdescriptionに「この要件を満たすために使う」のような記述があると有用:
```yaml
description: |
  growth_logのセンサーJSON生成が必要な時。
  非機能要件「SDカード32GBに収まる」を前提としたストレージ設計時。
```
