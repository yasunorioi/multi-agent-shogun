## 1. 設計原則

### 1-1. Superpowersから取り入れた原則

| 原則 | 内容 | shogun適用 |
|------|------|-----------|
| **description = トリガー条件** | 「いつ発動するか」だけ書く。「何をするか」を書くとLLMがdescriptionだけ読んで本文を飛ばす | 全スキル適用 |
| **CSO（Claude Search Optimization）** | スキル発見のためのキーワードを意識的に配置。ユーザーが使いそうな動詞・名詞をdescriptionに含める | 全スキル適用 |
| **トークン効率** | 頻繁ロード系は本文200語以下。コードスニペットは最小限 | 全スキル適用 |
| **名前参照のみ** | @リンクでコンテキスト消費しない。関連スキル名を記載するだけ | 全スキル適用 |
| **TDDアプローチ** | RED(失敗するテスト)→GREEN(最小実装)→REFACTOR | コード生成系のみ |

### 1-2. shogun向けカスタマイズ

| 原則 | 内容 |
|------|------|
| **エージェント層の明示** | frontmatterに `agent` フィールド。どの階層が使うか明示 |
| **ペルソナ/スキル分離** | 戦国口調はスキルに含めない。コードとドキュメントの品質はペルソナに依存しない |
| **過剰設計の排除** | Superpowersの圧力テスト・LLM固有最適化等は不採用。Simple > Complex |

## 2. Frontmatter仕様

```yaml
---
name: skill-name          # kebab-case、必須、一意
description: |             # 必須、max 1024 chars
  いつ発動するか。ユーザー/エージェントが使いそうなキーワードを含める。
  「何をするか」ではなく「どういう状況で必要になるか」を書く。
agent: 足軽               # 推奨実行エージェント（将軍/家老/足軽/部屋子/お針子）省略時=足軽
---
```

### フィールド定義

| フィールド | 必須 | 型 | 制約 | 備考 |
|-----------|:----:|-----|------|------|
| `name` | ✓ | string | kebab-case, 一意 | ファイル名と一致推奨 |
| `description` | ✓ | string | max 1024 chars | **トリガー条件のみ**。CSO意識 |
| `agent` | - | string | 将軍/家老/足軽/部屋子/お針子 | 省略時は足軽 |

### 旧TEMPLATE.mdから削除したフィールド

| 削除フィールド | 理由 |
|--------------|------|
| `trigger` | descriptionに統合（重複排除） |
| `target_path` | 本文のContextセクションで十分 |
| `needs_audit` | 運用判断であり、スキル定義に含めるべきでない |
| `output_format` | 本文のInstructionsで記述すれば十分 |
| `arguments` | 本文のUsageセクションで記述すれば十分 |
| `agent_type` | `agent` に簡略化・リネーム |

## 3. 本文構成

```markdown
# {Skill Name}

{1-2文の概要。何をするか。}

## When to Use

- トリガー条件1（状況の記述）
- トリガー条件2

## Instructions

### Step 1: ...
### Step 2: ...

## Examples（省略可）

## Notes（省略可）
```

### セクション定義

| セクション | 必須 | 目安語数 | 内容 |
|-----------|:----:|---------|------|
| 見出し + 概要 | ✓ | 1-2文 | 何をするか |
| When to Use | ✓ | 3-5項目 | いつ使うか（descriptionの詳細版） |
| Instructions | ✓ | 主要部分 | 手順。コードスニペットは最小限（10行以内/ブロック） |
| Examples | - | 1-2例 | 入出力例。省略可 |
| Notes | - | 3-5項目 | 注意点・関連スキル名。省略可 |

### トークン予算

| スキル種別 | 本文上限 | 理由 |
|-----------|---------|------|
| 頻繁ロード系（getting-started等） | **150語以下** | コンテキスト節約 |
| 標準スキル | **200語以下** | Superpowers推奨に準拠 |
| コード生成系（テンプレート含む） | **400語以下** | コードブロック分の余裕 |

## 4. description の書き方（CSO）

### NG例（「何をするか」を書いている）

```yaml
description: 農業用センサーデータ分析基盤をSQLiteで構築。DIFF計算、時間帯平均、値積算、日射比例灌水などの時間関係機能をSQL集計で実現。
```

LLMはこのdescriptionを読んで「何をするか分かった」と判断し、本文のInstructionsを飛ばす。

### OK例（「いつ発動するか」を書いている）

```yaml
description: |
  SQLiteで農業センサーのDIFF計算・日射積算・灌水トリガーを作る時。
  UECS CCMデータの時系列分析、日中/夜間分離、ストレス時間集計が必要な時。
```

### CSO チェックリスト

descriptionに以下を含めよ:
- [ ] ユーザーが検索しそうな**動詞**（作る、計算する、設定する、デプロイする）
- [ ] 対象の**技術名**（SQLite、MQTT、GPIO、UECS等）
- [ ] **状況**を示す語句（「〜する時」「〜が必要な時」「〜を受けた時」）

## 5. 旧フォーマットからの移行ガイド

### 移行手順

1. frontmatterを `name` + `description` + `agent`（任意）に縮小
2. descriptionを「いつ発動するか」に書き換え（CSO意識）
3. 本文を200語以内に圧縮:
   - 巨大なコードブロック → 骨格のみ残し、詳細コメントで補完
   - 複数の実装パターン → 最も汎用的な1パターンのみ
   - Node-RED連携・cron設定等の付随情報 → Notes に1行で参照先記載
4. 関連スキルの `[link](./path)` → 名前のみ記載（`関連: uecs-ccm-monitor`）

### 移行チェックリスト

- [ ] frontmatterが3フィールド以内
- [ ] descriptionに「何をするか」が含まれていない
- [ ] 本文が400語以下
- [ ] コードブロック1つあたり10行以内
- [ ] 戦国口調が混入していない

## 6. ディレクトリ構成

### 単一ファイルスキル（推奨）

```
skills/
├── sqlite-agri-analytics.md
├── emergency-sensor-handler.md
└── ...
```

### ディレクトリスキル（スクリプト付きの場合のみ）

```
skills/
└── skill-name/
    ├── SKILL.md          # エントリーポイント（必須）
    └── scripts/          # 補助スクリプト（省略可）
```

**原則**: 単一ファイルで収まるなら単一ファイル。ディレクトリ化は補助ファイルがある場合のみ。
