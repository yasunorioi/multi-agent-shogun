## 1. 背景と動機

### 1.1 現行アーキテクチャ (v3.x) の到達点と限界

shogun v3.x は戦国軍制メタファーで7エージェントの並列開発基盤を実現した。
しかし、PDCAサイクルの自動回転に向けた最後のピース — **AI同士の通信解禁** — が未実装のまま残っている。

**現行の強み（残すべきもの）:**
- 没日録DB（正データ永続化）
- お針子監査（事後品質ゲート）
- YAML+send-keys通信（全ログ人間可読）
- 権限マトリクス（DB読み書き権限の集約）
- bloom_routing（L4-L6は軍師委譲）
- Preflight Check（上流品質作り込み）

**現行の限界（解決すべきもの）:**

| 課題 | 根本原因 | 影響 |
|------|---------|------|
| PDCAが人間ボトルネック | AI→人間→AI の伝言ゲーム | 回転速度の天井 |
| 老中の過負荷 | 全通信が老中を経由 | 老中コンテキスト枯渇 |
| 合議の場がない | 1対1通信のみ（YAML inbox） | 多角的検証が困難 |
| 支店（OMC等）の成果物検証が手狭 | 本店tmux内で全てやろうとする | スケール限界 |

### 1.2 ハーネスエンジニアリングの枠組み

Mitchell Hashimoto提唱のハーネスエンジニアリング4要素で再設計する:

| 要素 | 定義 | v3.xの状態 | v4.0で強化すべき点 |
|------|------|-----------|------------------|
| **アーキテクチャ制約** | やってはいけないことを仕組みで防ぐ | F001-F006、権限マトリクス | 支店のworktree隔離、納品IF |
| **フィードバックループ** | ミスを検出する仕組み | お針子事後監査 | 2F合議による多角検証、リアルタイム逸脱検知 |
| **ワークフロー制御** | 行動順序と範囲の設計 | bloom_routing、Preflight Check | 三階間のゲート設計 |
| **改善サイクル** | ハーネス自体を育てる | feedbackメモリ→ルール昇格 | ミスパターン→ルール自動昇格パイプライン |

### 1.3 POSIX対応表

AIエージェントシステム ≒ POSIX階層（殿の洞察）。50年の知見を再発明するな。

| POSIX概念 | v3.x対応 | v4.0対応 |
|-----------|---------|---------|
| process isolation | tmuxペイン | 1F: worktree隔離、2F: BBSスレッド隔離 |
| IPC | YAML inbox + send-keys | 1F→2F: 納品POST、2F内: BBSレス、2F→3F: サマリ |
| named pipe (FIFO) | 老中が手動で中継 | スレッド（point-to-point的な議題単位の通信路） |
| permission / capability | instructions + 権限マトリクス | 板のwriters制限 = POSIX ACL |
| syslog | 没日録DB | 没日録DB + BBS DAT = 二重ログ |
| signal | send-keys | send-keys（3Fのみ）+ BBS通知（2F以下） |
| cgroup / namespace | なし | 1F worktree = namespace、リソース上限 = cgroup |
| systemd unit | スキル(.md) | スキル + swarm.yamlのエージェント定義 |
| exit code | subtask status (done/failed) | 納品IFのtest_result (PASS/FAIL) |
| fork() | なし | worker_ctl.sh / 支店の動的起動 |
| message queue (mq_open) | なし | BBS板（1:N pub/sub。pipeよりmqが正確） |
| SELinux / AppArmor | F001-F006 | 板のwriters + 納品IFのスキーマ検証 |

### 2.1 各階の責務

**温度理論との対応（temperature理論）:**

三階建ては温室三層制御・Finance三層と同型。下層(t=0)が上層(t>0)を黙らせる構造。

| 階 | temperature | 対応 | 特性 |
|----|-------------|------|------|
| 3F 本店 | t ≈ 0 | Layer1: 爆発防止 / daily_risk.py | 不可逆判断。ルールベース。ハルシネーションしない |
| 2F 合議場 | t > 0 | Layer3: 知恵 / Crucix | 自由議論。創発。ハルシネーション前提で使う |
| 1F 支店群 | t = 0〜中 | Layer2: ガムテ修理 | 実装。型で締めた入出力。worktreeで隔離 |
| kenshu_gate | t = 0 | ゲート判定 | PASS/FAIL/CONDITIONALの3値。決定論的 |

**bloom_routingの発動階:** bloom判定は**3F（老中）**で実行する。L1-L3は1Fに直接配布、L4-L6は2F（軍師板）経由で分析後に配布。

#### 3F: 本店（Honten）

**役割**: 不可逆な判断のみを担う。

| 判断 | 実行者 | ゲート |
|------|--------|--------|
| git push (private) | 老中 | 殿の暗黙承認（dashboard確認） |
| git push (origin) | 殿 | 殿の明示的承認 |
| マージ（worktree→main） | 老中 | 2F合議PASS |
| プロジェクト方向性変更 | 将軍 | 殿との対話 |
| 新規PJ立ち上げ | 将軍 | 殿の裁定 |
| スキル化承認 | 将軍 | 設計書レビュー |

**通信制約:**
- 将軍→老中: YAML（従来通り）
- 老中→2F: BBS POST（検収スレに納品物を投げる）
- 老中←2F: BBS読み取り（合議結果を読む）
- 将軍⇔殿: 直接対話（最重要チャネル）
- **send-keysは3F内のみ使用**（2F以下にはBBS経由）

#### 2F: 合議場（Gougijo）

**役割**: AI同士の自由議論による多角的品質検証。

**新設の板:**

| 板名 | 目的 | writers | 読み取り |
|------|------|---------|---------|
| kenshu（検収板） | 1Fからの納品物レビュー | 全員 | 全員 |
| kenshu_gate | 合議結果（PASS/FAIL+理由） | お針子, 軍師, 老中 | 老中 |

**既存板の役割明確化:**

| 板名 | v4.0での役割 |
|------|-------------|
| ninmu（任務板） | 3F→1Fのタスク配布（老中が投稿） |
| kansa（監査板） | お針子のフル監査ログ |
| gunshi_board | 軍師の戦略分析 |
| zatsudan（雑談板） | PDCA振り返り・ナレッジ共有 |

**合議プロセス（検収フロー）:**

**Bloom別検収ファストトラック:**

| Bloomレベル | 検収方式 | 合議参加者 | kenshu_gate writer |
|------------|---------|-----------|-------------------|
| L1-L3 | ファストトラック | 老中スポットチェックのみ（2F合議スキップ） | 老中 |
| L4-L5 | 標準合議 | お針子 + 軍師（最低2名の実質的レビュー必須） | お針子, 軍師 |
| L6 | フル合議 | お針子 + 軍師 + 外部レビュア（将来） | お針子, 軍師 |

> L1-L3のファストトラックにより、2Fの負荷をL4-L6に集中させ、支店数増加時のボトルネックを回避。

**L4-L6 標準合議フロー:**

```
1F納品 → 検収板にスレ立て（納品物概要+diff+テスト結果）
  │
  ├→ お針子: ルーブリック監査 → レスで採点
  ├→ 軍師: 戦略適合性チェック → レスで評価
  ├→ (将来)勘定吟味役: 外部モデル監査 → レスで評価
  │
  ▼ 合議（スレ内議論）— 最低2名の実質的レビュー必須（「LGTM」のみは不可）
  │
  ▼ kenshu_gate に判定投稿
  │   PASS → 老中がマージ（3F）
  │   FAIL → 任務板にリジェクト通知POST → 1Fに差し戻し（§3.1逆通知経路参照）
  │   CONDITIONAL → 軽微修正を老中が実施してマージ
```

**合議不一致時のタイブレーカー:**
お針子PASSだが軍師FAIL（またはその逆）の場合:
1. 老中が検収板の議論を読み、3Fで裁定
2. 老中判断困難 → dashboard.md「🚨要対応」に記載し殿にエスカレーション

**AI通信解禁のルール:**
- 2F内のBBSレスは自由（全発言がDATに残る = 監査証跡）
- 板のwriters制限がアクセス制御（POSIX ACL相当）
- スレッドが議論の単位 = 名前付きパイプ
- 1スレ1議題（脱線は雑談板へ）

#### 1F: 支店群（Shiten）

**役割**: 実装とPDCA自律回転。暴走してもworktree内で閉じる。

**支店の種類:**

| 支店 | 実行環境 | 隔離方式 | 通信方式 |
|------|---------|---------|---------|
| tmux足軽 | ローカルtmux | git worktree | BBS POST |
| Codex足軽 | OpenAI Codex API | sandbox | webhook→BBS |
| MBP支店 | mbp.local SSH | 物理隔離 | SSH exec→BBS |
| (将来) Docker支店 | Docker container | namespace | API→BBS |

**支店の制約（ハーネス）:**
1. **mainブランチに触れない** — worktreeのブランチでのみ作業
2. **納品IFを通じてのみ上に送れる** — 直接マージ不可
3. **リソース上限** — (将来) cgroup的な制約（トークン上限、時間制限）
4. **支店内AI通信は自由** — ただしworktree内ファイルとBBSのみ

**納品インターフェース（1F→2F）:**

```yaml
# 検収板への投稿フォーマット
subject: "subtask_XXXX 納品: {1行サマリ}"
body: |
  ■ subtask_id: subtask_XXXX
  ■ cmd_id: cmd_YYY
  ■ worker: ashigaru1
  ■ branch: worktree-subtask-XXXX
  ■ diff_summary: {変更ファイル数, 追加行数, 削除行数}
  ■ test_result: {PASS/FAIL, テスト数, カバレッジ}
  ■ commit_hash: {最終コミットハッシュ}
  ■ self_review: {自己レビューコメント}
```

## 3. 通信設計

### 3.1 通信経路の全体図

```
殿 ⇔ 将軍(3F)    : 直接対話（stdin/stdout）
将軍 → 老中(3F)    : YAML + send-keys（従来通り）
老中 → 任務板(2F)  : BBS POST（タスク配布）
任務板 → 1F        : BBS通知（send-keys or notify_exec）
1F → 検収板(2F)    : BBS POST（納品）
検収板内(2F)        : BBS レス（AI自由合議）
2F → kenshu_gate   : BBS POST（判定結果）
老中 ← kenshu_gate : BBS READ（判定読み取り）
kenshu_gate FAIL    : → 任務板にリジェクト通知POST → 1Fにsend-keys/notify_exec  ★逆通知
老中 → 没日録DB    : kenshu_gate読み取り時に合議結果を自動記録（audit record）  ★書き戻し
老中 → main merge  : git merge（3Fでのみ実行）
老中 → dashboard   : dashboard.md更新（殿への報告）
```

**2F→1F逆通知経路（差し戻し）:**
kenshu_gateにFAILが投稿された場合:
1. 老中がkenshu_gateを読む
2. 任務板に「リジェクト通知」をPOST（差し戻し理由 + 修正ポイント）
3. 任務板の通知機能で1Fの担当足軽にsend-keys/notify_exec
4. 足軽は修正後、検収板に再納品

**没日録DBへの合議結果書き戻し:**
老中がkenshu_gateを読んだ時点で、没日録DBにaudit recordを記録する:
```bash
python3 scripts/botsunichiroku.py audit add subtask_XXX \
  --verdict PASS/FAIL/CONDITIONAL \
  --kenshu_thread YYY \
  --reviewers "ohariko,gunshi" \
  --summary "合議結果1行サマリ"
```
これにより「正データは没日録」原則がBBS合議導入後も維持される。

### 3.2 通信の型付け戦略

Y-Combinator論文の知見を適用: **足軽層は型を締め、将軍・軍師層は緩く保つ**。

| 通信 | 型の厳密さ | 理由 |
|------|-----------|------|
| 納品IF（1F→2F） | **厳格** | スキーマ検証で不完全な納品を弾く |
| 合議レス（2F内） | 緩い | 自由議論が創発を生む |
| 判定結果（2F→3F） | **厳格** | PASS/FAIL/CONDITIONALの3値 + 構造化理由 |
| タスク配布（3F→1F） | 中程度 | Preflight Checkで品質担保 |
| 殿⇔将軍 | 自由 | 壁打ちに型制約は邪魔 |

### 3.3 監査証跡の二重化

| レイヤー | 書き込み | 永続性 | 用途 |
|---------|---------|--------|------|
| 没日録DB | 老中（3F） | 永続 | 正データ。コマンド・サブタスク・日記 |
| BBS DAT | 全エージェント | 永続 | 議論ログ。合議プロセスの証跡 |

**相互参照:**
- 没日録: `subtask_XXX` → BBS: `検収板スレ#YYY`
- BBS: `>>subtask_XXX` → 没日録のsubtask参照

## 4. PDCA自動回転

### 4.1 自動化されるPDCAサイクル

```
Plan:  軍師が戦略分析 → 軍師板に投稿
  │
  ▼
Do:    足軽(1F)が実装 → worktreeで作業 → 検収板に納品
  │
  ▼
Check: 2F合議 → お針子監査 + 軍師適合性 → kenshu_gate判定
  │
  ├─ PASS → Act へ
  └─ FAIL → 検収板に差し戻し理由 → 足軽が修正 → 再納品（Do に戻る）
  │
  ▼
Act:   老中がマージ(3F) → 雑談板でPDCA振り返り → feedbackメモリ更新
       → 次のPlan へ
```

### 4.2 人間介入ポイント（ゲート）

PDCAを自動回転させつつ、不可逆な判断には人間ゲートを設ける:

| ゲート | 介入者 | 条件 |
|--------|--------|------|
| git push (origin) | 殿 | 常に |
| プロジェクト方向性 | 殿 | 常に |
| PDCA 3回FAIL | 殿 | エスカレーション（既存ルール） |
| Bloom L6タスク | 殿 | 戦略・創造は殿裁定（既存ルール） |
| 新規PJ/スキル化 | 殿 | 常に |
| git push (private) | 老中 | 自動（殿の暗黙承認） |
| マージ | 老中 | 2F合議PASS後 |

### 4.3 暴走防止策

| リスク | 対策 | POSIX対応 |
|--------|------|-----------|
| 1Fの無限ループ | 時間制限 + トークン上限 | ulimit |
| 2Fの議論空転 | スレ最大レス数制限（>>50で強制判定） | SIGALRM |
| 合議の全員賛成バイアス | 勘定吟味役（別モデル）の導入 | 別UID |
| 納品IFスキーマ違反 | 投稿時の構造検証 | input validation |
| 1Fがmainを汚染 | worktree強制 + mainへの直接commit禁止 | chroot |

## 5. ハーネス強化計画

### 5.1 アーキテクチャ制約（仕組みで防ぐ）

| 制約 | 実装方法 | 対象階 |
|------|---------|-------|
| main直接commit禁止 | git hookまたはBBS gate | 1F |
| 納品IFスキーマ検証 | BBS POST時にバリデーション | 1F→2F |
| 板のwriters制限 | swarm.yaml ACL | 2F |
| マージ権の集約 | 老中のみ実行可 | 3F |

### 5.2 フィードバックループ（ミスを検出する）

| ループ | v3.x | v4.0 |
|--------|------|------|
| 事後監査 | お針子（単独） | お針子 + 軍師 + 外部（合議） |
| ミスパターン検出 | 人間が気づく | BBS DATから自動抽出 |
| 逸脱検知 | なし | 納品IF違反の即時検出 |
| 品質トレンド | なし | (将来) 検収PASS率の推移 |

### 5.3 ワークフロー制御（行動順序の設計）

```
タスク受領(3F) → bloom判定 → Preflight Check → 配布(任務板)
  → 実装(1F) → 自己テスト → 納品(検収板)
  → 合議(2F) → 判定(kenshu_gate) → マージ(3F)
```

### 5.4 改善サイクル（ハーネス自体を育てる）

**ミスパターン→ルール昇格パイプライン:**

```
1. 検収FAIL → 差し戻し理由がBBSに記録
2. 同一パターンのFAILが3回発生 → お針子が検出
3. お針子が「ルール昇格候補」を監査板に投稿
4. 軍師が汎用性を評価 → 軍師板にレス
5. 老中が承認 → Preflight Checkに新項目追加 or スキル化
6. (L6相当) 殿の承認が必要な場合 → dashboard.md 🚨要対応
```

これがHashimotoの言う「ミスを犯したとき、二度と起きないよう仕組みで解決する」の実装。

## 8. 設計原則（変わらないもの）

これらは三階建てになっても不変:

1. **マクガイバー精神** — シンプル・ローコスト・手元の道具で解決
2. **下層が上層を黙らせる** — 温室三層制御と同じ。worktree隔離(1F)が合議(2F)を守り、合議が本店(3F)を守る
3. **正データは没日録DB** — BBSは補助。矛盾時はDB優先
4. **殿は壁打ち相手** — 将軍の最大価値は対話。自動化しても殿との対話チャネルは最優先
5. **怒り駆動開発** — ミスパターンが怒りとなり、ルールに昇格する。ハーネスの燃料は怒り
6. **無知の知** — 完璧な自動化は幻想。人間ゲートは保険ではなく必須
7. **50年の知見を借りろ** — POSIX対応表を確認してから設計せよ
